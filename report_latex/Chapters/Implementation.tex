\section{Operational Pipeline (Current)}
%==============================================================================

The current end-to-end workflow is implemented as numbered scripts with explicit artifacts:
\begin{enumerate}\tightlist
    \item \code{00\_prepare\_hour0\_data.py} and \code{00b\_fit\_hour0\_preprocessor.py}
          prepare and encode hour-0 training data.
    \item \code{00c\_train\_hour0\_model.py} trains an IID generator.
    \item \code{01\_preprocess\_data.py} creates autoregressive tabular rows and carves patient-disjoint train / test / holdout splits (\code{autoregressive\_data.csv}, \code{real\_test.csv}, \code{real\_holdout.csv}).
    \item \code{01b\_fit\_preprocessor.py} fits the full-data autoregressive preprocessor on the training cohort.
    \item \code{02\_train\_model.py} trains the autoregressive generator.
    \item \code{03\_generate\_synthetic.py} loads model artifacts and exports synthetic samples.
    \item \code{run\_sweep.py} performs unified hour-0 + autoregressive hyperparameter search.
\end{enumerate}

The architecture is explicitly two-stage and backbone-selectable (\code{forest-flow} or \code{hs3f}, default \code{hs3f}). Standalone evaluation scripts (\code{04\_evaluate\_quality.py}, \code{05\_evaluate\_hour0\_model.py}) are also available for ad-hoc inspection of individual outputs; these are described in the User Guide appendix. In the sweep-centric workflow above, quality and TSTR evaluation are integrated directly into \code{run\_sweep.py}.

\section{Training Implementation Details}
%==============================================================================

\subsection{Hour-0 IID Training}

\code{00c\_train\_hour0\_model.py} loads \code{models/hour0\_preprocessor.pkl}, transforms all hour-0 rows, and fits an IID generator:
\begin{equation}
\hat{p}(\mathbf{z}_0), \quad \mathbf{z}_0=[\mathbf{s},\mathbf{x}_0].
\end{equation}

The script persists model metadata including \code{model\_type}, training hyperparameters, and preprocessor linkage.

\subsection{Autoregressive Training}

\code{02\_train\_model.py} loads \code{models/preprocessor\_full.pkl}, transforms selected rows of \code{autoregressive\_data.csv}, and reconstructs split matrices via persisted indices:
\begin{equation}
X_{\text{target}} = X[:, \text{target\_indices}], \quad
X_{\text{condition}} = X[:, \text{condition\_indices}].
\end{equation}

This removes the previous dependence on naive prefix slicing and is needed in the current implementation when transformed columns are type-grouped.

\subsection{Categorical Typing Interface}

Both backbones receive \code{feature\_types} from preprocessor metadata:
\begin{itemize}\tightlist
    \item \code{'q'} quantitative,
    \item \code{'c'} categorical.
\end{itemize}

Forest-Flow internally overrides noised target dimensions to quantitative mode (flow arithmetic constraint), while HS3F keeps explicit categorical heads \cite{akazan2024hs3f}.

\subsection{Categorical Labels and Flow Integration (Codebase)}

For HS3F categorical dimensions, training remaps observed categories to contiguous local class indices for XGBoost multiclass fitting; \code{sample()} inverts that map via stored \code{local\_to\_orig} so synthetic rows use the same encoded IDs as the preprocessor artifact.
The training remap yields contiguous class indexing under arbitrary subsampling without sparse multiclass labels at fit time.

For continuous targets under Forest-Flow, sampling follows explicit first-order integration.
Forest-Flow integrates the backward flow with explicit Euler updates (\code{X\_t += h * Y\_hat}) in \code{src/synth_gen/models/forest\_flow.py}.
HS3F continuous dimensions default to \code{solver="rk4"} in \code{src/synth_gen/models/hs3f.py}, with \code{solver="euler"} available as an alternative.

%==============================================================================
\section{Sampling and Reconstruction}
%==============================================================================

\subsection{Model-Space Sampling}

For conditional generation, models sample:
\begin{equation}
\tilde{X}_{\text{target}} \sim p_{\theta}(X_{\text{target}} \mid X_{\text{condition}}).
\end{equation}

In sweep mode, conditioning rows are sampled from the real transformed condition matrix and reused as context for synthetic targets.

\subsection{Rebuilding Full Feature Vectors}

The implementation reconstructs full transformed rows in preprocessor order:
\begin{equation}
X_{\text{full}}[:,\text{target\_indices}] \leftarrow \tilde{X}_{\text{target}}, \quad
X_{\text{full}}[:,\text{condition\_indices}] \leftarrow X_{\text{condition}}.
\end{equation}

Then \code{inverse\_transform()} restores tabular values and category labels.

\subsection{Post-Decode Clinical Normalization}

Known intervention indicators (e.g., \code{vent}, \code{vaso}) are explicitly clipped/rounded to $\{0,1\}$ before evaluation/export.
This step is intended to restore discrete $\{0,1\}$ semantics for binary treatment flags after decode.

\section{Evaluation Stack}
%==============================================================================

\subsection{Cohort-Level Data Splits}
\label{sec:cohort-splits}

All utility, fidelity, and privacy metrics are computed against real-data cohorts that are patient-disjoint from the generator's training cohort.
At preprocessing time (\code{scripts/01\_preprocess\_data.py}), the autoregressive table is partitioned at the \code{subject\_id} level into three CSVs with an 80\,/\,10\,/\,10 split drawn from a fixed RNG seed so the partition is reproducible across runs:
\begin{itemize}\tightlist
    \item \code{data/processed/autoregressive\_data.csv} -- the training cohort (80\,\% of patients), the only file the generator ever sees.
    \item \code{data/processed/real\_test.csv} -- a TSTR / fidelity reference cohort (10\,\% of patients).
    \item \code{data/processed/real\_holdout.csv} -- a privacy-evaluation holdout cohort (10\,\% of patients).
\end{itemize}

Because the split is performed on \code{subject\_id}, no patient's timesteps ever span the training cohort and an evaluation cohort.
Consequently:
\begin{itemize}\tightlist
    \item Multi-task TSTR (\S~\textbf{Multi-Task TSTR}) trains classifiers on synthetic data and tests them on sequences drawn from \code{real\_test.csv}; the real-trained baseline classifier is also fit and evaluated entirely within this disjoint cohort via an internal stratified split (\code{test\_size=0.3}).
    \item The distributional fidelity metrics (KS, correlation Frobenius norm, range-violation rate) are computed against a sample of \code{real\_test.csv} rather than the training matrix, so they measure distributional recovery of \emph{unseen} rows rather than reproduction of the training set.
    \item The DCR overfitting-protection and DOMIAS-style membership-inference attack use \code{real\_holdout.csv} directly as the holdout reference, giving the overfitting-vs-generalisation comparison a meaningful signal even when the generator is expressive enough to memorise.
\end{itemize}

The split is materialised once per preprocessing run and referenced explicitly by \code{AutoregressiveInputs.real\_test\_path} and \code{AutoregressiveInputs.real\_holdout\_df} in \code{src/synth_gen/sweep/data\_prep.py}, ensuring the sweep driver, the matched backbone-comparison driver, and any future evaluation scripts consume byte-identical reference cohorts.

\subsection{Multi-Task TSTR}

The active sweep uses a unified patient-level TSTR framework with two tasks:
\begin{enumerate}\tightlist
    \item Mortality classification (binary, patient-level),
    \item ICU LOS category classification (multi-class, patient-level).
\end{enumerate}

Each task uses a \code{\_PaddedSequenceClassifier}: patient trajectories are grouped by \code{subject\_id}, sorted by \code{hours\_in}, padded to a common length with a binary mask, flattened, and fed to a logistic-regression baseline. This patient-level design eliminates within-patient data leakage and captures the temporal structure the autoregressive generator is designed to preserve. The design evolution from the original row-level three-task framework and the rationale for removing vasopressor-use prediction are documented in the Evaluation chapter. The TSTR paradigm follows established synthetic health-data evaluation practice \cite{esteban2018rcgan}.

\subsection{Task Metrics}

For binary tasks, reported metrics include accuracy, F1, and ROC-AUC.
For LOS (multi-class), macro-F1 and one-vs-rest macro ROC-AUC are included.

\subsection{Distributional Quality Metrics}

In addition to task utility, sweep evaluation computes:
\begin{enumerate}\tightlist
    \item Average KS statistic across valid numeric columns,
    \item Correlation matrix Frobenius norm,
    \item Clinical range violation percentage.
\end{enumerate}

Let $\mathcal{J}$ be valid numeric columns:
\begin{equation}
\text{KS}_{\text{avg}} = \frac{1}{|\mathcal{J}|}\sum_{j\in\mathcal{J}}\sup_x |F_j^{\text{real}}(x)-F_j^{\text{synth}}(x)|.
\end{equation}
\begin{equation}
\text{CorrDiff} = \|C^{\text{real}} - C^{\text{synth}}\|_F.
\end{equation}

Columns with insufficient finite samples or zero variance are excluded before these computations to avoid undefined correlations.

\subsection{Privacy Risk Metrics}

To complement utility/fidelity metrics, the current implementation adds four privacy checks used in recent tabular-synthetic evaluation practice \cite{lautrup2024syntheval,breugel2023domias}:
\begin{enumerate}\tightlist
    \item Distance to Closest Record (DCR) summary ($\mathrm{median}$, tails, exact-match rate),
    \item DCR baseline-protection score (synthetic-vs-random closeness to real data),
    \item DCR overfitting-protection score (synthetic closeness to train vs holdout),
    \item DOMIAS-inspired membership inference attack (MIA) ROC-AUC and attacker advantage.
\end{enumerate}

For synthetic samples $\tilde{x}_i$ and real training set $\mathcal{D}_{\text{train}}$, DCR is:
\begin{equation}
\mathrm{DCR}(\tilde{x}_i,\mathcal{D}_{\text{train}})=\min_{x\in\mathcal{D}_{\text{train}}}\|\phi(\tilde{x}_i)-\phi(x)\|_2,
\end{equation}
with mixed-type embedding $\phi(\cdot)$ (min-max normalized numerics + one-hot categoricals).
Baseline protection follows the SDMetrics ratio form \cite{sdmetricsPrivacyDocs}:
\begin{equation}
\mathrm{Score}_{\text{baseline}}=\min\left(1,\frac{\mathrm{median}\,\mathrm{DCR}_{\text{synth}\rightarrow\text{train}}}{\mathrm{median}\,\mathrm{DCR}_{\text{random}\rightarrow\text{train}}}\right).
\end{equation}

Overfitting protection compares train-vs-holdout nearest-neighbor proximity for synthetic rows, where the holdout is the patient-disjoint \code{real\_holdout.csv} cohort defined in \S\ref{sec:cohort-splits}:
\begin{equation}
\mathrm{Score}_{\text{overfit}}=\min\left(1,\,2\left(1-\Pr\left[\mathrm{DCR}_{\text{synth}\rightarrow\text{train}} < \mathrm{DCR}_{\text{synth}\rightarrow\text{holdout}}\right]\right)\right).
\end{equation}

Membership risk is quantified with a DOMIAS-style local overfitting score \cite{breugel2023domias}:
\begin{equation}
s(x)=\log\frac{d_{\text{ref}}(x)+\epsilon}{d_{\text{synth}}(x)+\epsilon},
\end{equation}
where $d_{\text{synth}}$ is nearest-neighbor distance to synthetic data and $d_{\text{ref}}$ is distance to the patient-disjoint holdout cohort, so a non-zero attacker advantage can only reflect closeness to training patients rather than to the evaluation cohort itself.
Attack performance is reported via ROC-AUC (and attacker advantage) in the membership-inference paradigm \cite{shokri2017membership}.

\paragraph{Interpretation guardrail.}
DCR-style proximity metrics are useful diagnostics but should not be treated as sufficient legal/anonymity evidence on their own; recent work shows distance-only proxies can miss true membership leakage, so MIA results remain primary for privacy claims \cite{yao2025dcrdelusion}.

\subsection{Empirical Outcomes from the Hyperparameter Sweep}

The canonical log \code{results/sweep\_results.csv} covers a $5 \times 8 = 40$-cell grid over $n_t \in \{1,3,5,7,9\}$ and $n_{\text{noise}} \in \{1,3,5,7,9,11,13,15\}$.
At the time of writing, $35$ cells are complete under the single-seed HS3F sweep and the remaining $5$ cells at $n_t=9$, $n_{\text{noise}} \in \{7,9,11,13,15\}$ are in progress; figures in this subsection therefore describe $35/40$ of the grid.
Seed-level uncertainty columns (\code{*\_std}, \code{*\_ci95}, \code{trajectory\_seed\_count}) are part of the \code{SWEEP\_RESULT\_COLUMNS} schema but are populated only in the matched-cell backbone comparison (\S\ref{sec:backbone-comparison}); the main-grid results below are single-seed point estimates.
When the multi-seed protocol is active, each trajectory-sampling seed in \code{TSTR\_TRAJECTORY\_SAMPLING\_SEEDS} also drives the evaluator \code{random\_state} (real/synthetic train-test split, real-baseline classifier fit, and privacy-attack subsampling), so the reported \code{\_std} and \code{\_ci95} columns capture both trajectory-sampling variance and evaluator variance rather than trajectory-sampling variance alone.
The following subsections present findings across four dimensions: downstream utility (TSTR), distributional quality, computational cost, and extrapolation analysis.
All figures are generated by \code{scripts/plot\_sweep\_results.py} from the canonical sweep CSV.

    \item Very low $n_t$ (e.g., $n_t=1$) consistently yields the poorest utility, with ROC-AUC in the $0.49$--$0.57$ range.
    \item Increasing $n_{\text{noise}}$ beyond $\approx 9$ provides diminishing or negative returns for most $n_t$ values, suggesting overfitting of the noise-augmented training signal.
    \item The line charts reveal high variance across configurations, with no single monotonic trend---indicating that the optimal $(n_t, n_{\text{noise}})$ pair is dataset-specific and not simply ``more is better.''
\end{itemize}

These metrics reveal a diagnostic asymmetry:
\begin{itemize}\tightlist
    \item \textbf{Accuracy} is strongly sensitive to $n_t$: low $n_t$ produces near-random predictions ($< 0.20$), while $n_t \geq 5$ recovers accuracy to $0.86$--$0.90$, approaching the real-model baseline. This suggests that sufficient time-level discretization is necessary for the flow to capture the dominant class distribution.
    \item \textbf{F1} remains persistently low ($0.05$--$0.30$) even at the best configurations, with a substantial gap to the real model ($\approx 0.51$). \end{itemize}

\subsubsection{Distributional Quality Metrics}

Figure~\ref{fig:sweep-quality} presents the three quality metrics across the sweep grid.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{sweep_quality_metrics.png}
\caption{Quality metrics across the $(n_t, n_{\text{noise}})$ grid. Left: average KS statistic (lower is better). Centre: correlation Frobenius norm (lower is better). Right: clinical range violation percentage (lower is better). Higher $n_t$ generally improves all three metrics.}
\label{fig:sweep-quality}
\end{figure}

\begin{itemize}\tightlist
    \item \textbf{KS statistic.} Mean KS ranges from $0.736$ to $0.779$ across the completed grid cells. These are high absolute values: a mean KS of $\sim$0.75 indicates the synthetic marginal distributions are substantially displaced from the real ones across many features, well above the $0.10$--$0.20$ range typical of high-quality tabular generators in published benchmarks \cite{yan2022benchmarking}. Two factors partially explain the magnitude. First, the numerics-only average is computed on raw (unscaled) features whose marginals may differ sharply in location and scale from the unit-interval domain where KS is most sensitive; preprocessing artefacts and rare-value spikes in ICU measurements inflate the statistic. Second, binary intervention flags (e.g., ventilation, vasopressor) are transported as continuous flows during generation and clipped post-hoc, producing non-binary intermediate values that increase per-feature KS before rounding. Despite these mitigations, the values confirm that marginal fidelity remains a principal limitation of the current system. There is a weak trend toward lower KS at higher $n_t$ ($\approx 0.743$--$0.757$ at $n_t=9$ vs.\ $\approx 0.736$--$0.738$ at $n_t=1$); the small dynamic range suggests that marginal alignment is not primarily governed by time-level discretisation but by representational or conditioning constraints in the autoregressive scheme.
    \item \textbf{Correlation Frobenius norm.} The norm ranges from $\approx 2.30$ to $3.26$, with the minimum at $(n_t{=}3, n_{\text{noise}}{=}3)$ ($\approx 2.295$). Unlike accuracy and range violations---which improve monotonically with $n_t$---correlation-Frobenius values are lowest along the $n_t=3$ row and remain in a narrow $2.41$--$2.63$ band for $n_t \in \{5,7,9\}$, with $n_t=1$ noticeably worse ($2.66$--$3.26$). Inter-feature dependency structure is therefore already well-recovered at moderate time discretisation and does not visibly sharpen further as $n_t$ increases; the gains from finer $n_t$ observed elsewhere are carried by range fidelity and utility rather than by the correlation-Frobenius axis.
    \item \textbf{Clinical range violations.} Range violations drop substantially at higher $n_t$: at $n_t \geq 7$, most configurations achieve $\approx 14$--$19\%$ violations, compared with $\approx 15$--$29\%$ at $n_t \leq 3$. This represents a meaningful improvement and indicates that finer time discretization helps the flow dynamics better respect the support of the training distribution. Individual outliers exist (e.g., $9.33\%$ at one low-$n_t$ configuration), but the dominant trend favours higher $n_t$.
\end{itemize}

Combined with the quality and utility trends above, the most cost-effective configurations appear to lie in the $n_t \in \{5,7\}$, $n_{\text{noise}} \in \{5,9\}$ region, where utility and quality gains plateau before computational cost escalates sharply.

These trends indicate that the remaining utility gap is structural rather than a hyperparameter tuning artefact: the generator's representational capacity or the autoregressive factorisation itself may need architectural improvements to close the gap.

\paragraph{What the matched cell says about the CTGAN baseline.}
CTGAN's mortality ROC-AUC of $0.497$ is indistinguishable from a constant-score classifier---the standard interpretation of AUROC $\approx 0.5$---and its balanced accuracy ($0.507 \pm 0.020$) is similarly near chance. In the raw-accuracy column of the underlying CSV, CTGAN scores $0.622$, essentially tied with ForestFlow's $0.637$; however, under MIMIC's mortality class imbalance this raw figure mostly reflects majority-class (survived) prevalence rather than discrimination, which is exactly why Table~\ref{tab:backbone-matched} reports the balanced form. Once the majority-class bias is removed, CTGAN's classifier is no better than chance at ranking patients by mortality risk, despite its average-KS being the best of the three ($0.686$). The baseline is therefore failing for the right reason: it reproduces approximately-correct feature marginals (consistent with its best-in-class KS) but does not preserve the lagged-feature-to-outcome dependency structure a downstream classifier needs. This is consistent with the adapter being trained on the transformed joint table and sampled without per-row autoregressive conditioning, as flagged in the \S\textbf{Baseline Scope and Interpretation} disclaimer. The trajectory-level metrics reinforce this reading: CTGAN's transition smoothness ratio ($9.86$) is roughly double that of the tree-based backbones, so hour-to-hour transitions are correspondingly jumpier---a direct consequence of the missing per-row conditioning. The baseline is therefore informative in exactly the way a baseline should be: it demonstrates that the utility signal captured by patient-level TSTR ROC-AUC (and, equivalently, by balanced accuracy once calibration noise is absorbed into score ranking) is not trivially attainable by a generator that merely produces near-correct marginals.

\paragraph{CTGAN hyperparameters and fairness of the comparison.}
The CTGAN baseline is configured with the hyperparameters reported in the original paper \cite{xu2019ctgan} and exposed as library defaults in the \code{ctgan} package: a two-layer $256\text{-}256$ generator MLP, a two-layer $256\text{-}256$ discriminator MLP, embedding dimension $128$, $300$ training epochs, batch size $500$, Adam learning rate $2 \times 10^{-4}$ with weight decay $10^{-6}$, and PAC size $10$ (see \code{scripts/run\_sweep.py}, \code{CTGAN\_DEFAULT\_*}, exposed to \code{run\_matched\_backbone\_comparison.py} through the \code{--ctgan-*} CLI flags). Two small reproducibility points are worth noting: (i) when the requested PAC does not divide the batch size the adapter silently falls back to the largest valid divisor (no effect at $(500, 10)$), and (ii) PyTorch and OMP thread pools are pinned to a single thread via \code{torch.set\_num\_threads(1)} and \code{OMP\_NUM\_THREADS=1} so the training-time comparison is not biased by CTGAN exploiting more CPU threads than the XGBoost-based backbones. No CTGAN-specific tuning was attempted beyond the published defaults, so the comparison is against a standard out-of-the-box baseline rather than a hand-tuned one; if anything, this biases the comparison in CTGAN's favour by removing the possibility that the observed collapse is caused by hyperparameter misconfiguration.

\paragraph{Compute cost versus utility.}
The most striking feature of the matched-cell comparison is the inversion between computational cost and downstream utility. CTGAN is the most expensive model to train by a large margin: $8017$~s total wall-clock at $(n_t{=}5, n_{\text{noise}}{=}5)$, compared with $3224$~s for ForestFlow ($\approx 2.5\times$ slower than ForestFlow) and $2194$~s for HS3F ($\approx 3.7\times$ slower than HS3F). Despite consuming more compute than the two tree-based backbones combined, CTGAN produces the worst downstream utility on every TSTR axis examined at this cell: mortality ROC-AUC ($0.497$, significantly below HS3F's $0.672$ at three seeds), LOS macro-F1 ($0.173$, roughly $40\%$ lower than the $\approx 0.28$ achieved by both tree-based backbones), and transition smoothness ratio ($9.86$, roughly $2\times$ the HS3F/ForestFlow value). Balanced mortality accuracy is numerically close across all three backbones ($0.501$--$0.535$), but this near-chance clustering is itself the critique: the extra CTGAN wall-clock buys no measurable discrimination improvement over the cheaper tree-based alternatives. The longer training time is therefore not purchasing any measurable utility advantage. This reinforces rather than qualifies the ``Why HS3F'' decision developed below: HS3F wins the matched cell on ranking utility, trajectory coherence, training cost, and privacy parity, and the closest neural baseline at standard hyperparameters loses decisively on every utility axis despite consuming nearly four times the wall-clock of HS3F.

\paragraph{Interpreting the HS3F--ForestFlow discrimination profile.}
On the balanced-accuracy row of Table~\ref{tab:backbone-matched}, all three backbones cluster near chance: HS3F $0.501 \pm 0.012$, CTGAN $0.507 \pm 0.020$, ForestFlow $0.535 \pm 0.048$. The nominal ForestFlow advantage overlaps HS3F's confidence interval and is not statistically significant at three seeds. On ROC-AUC the ordering reverses---HS3F $0.672$ vs.\ ForestFlow $0.619$---and with CI95 half-widths of $\approx 0.021$ (HS3F) and $\approx 0.077$ (ForestFlow) from \code{backbone\_matched\_comparison.csv} the tree-based gap is again not statistically significant, while the HS3F--CTGAN gap on ROC-AUC is. The honest reading of the mortality row is therefore: (1) no backbone produces a downstream classifier that reliably discriminates mortality under the balanced-accuracy lens; (2) HS3F's ROC-AUC lead over CTGAN is real, its ROC-AUC lead over ForestFlow is suggestive but not significant at three seeds; (3) LOS macro-F1 is statistically tied between HS3F and ForestFlow ($0.284$ vs.\ $0.283$) and significantly above CTGAN ($0.173$). Patient-level utility at this cell is therefore an \emph{incremental} win for HS3F on ranking metrics rather than a categorical one; the decisive separation between HS3F and ForestFlow appears in the trajectory-level metrics discussed next.

\paragraph{Where HS3F does separate from ForestFlow: temporal coherence.}
The matched-cell trajectory-level metrics in Table~\ref{tab:backbone-trajectory} separate the two tree-based backbones more decisively than the patient-level TSTR row does. HS3F's autocorrelation distance ($0.22$) is roughly half of ForestFlow's ($0.44$), and its temporal cross-feature correlation drift ($0.26$) is likewise the lowest of the three. Both metrics are direct measurements of whether per-patient lag-1 temporal structure is preserved, both are computed on the same matched-cell synthetic cohort, and both use the same three-seed averaging, so the comparison is not confounded by sampling differences. The plausible mechanism follows from the backbone design. HS3F generates intervention flags (e.g.\ ventilation, vasopressors) and categorical admission variables through explicit classifier heads, so at step $j+1$ of the feature-wise autoregressive chain the conditioning vector contains cleanly-typed previously-generated features; ForestFlow transports the same discrete features as continuous flows and rounds post-hoc, so the conditioning vector that drives transition generation contains continuous intermediate values that round-trip through the discretization boundary. A $2\times$ increase in autocorrelation distance for ForestFlow, measured on identical patients and seeds, is exactly the signal the design hypothesis predicts this round-trip would produce. This is the strongest positive evidence for HS3F's architectural choice on our data.

\paragraph{Privacy verdict.}
All three generators achieve DCR baseline-protection $= 1.0$ under the SDMetrics ceiling, zero exact-match synthetic records (\code{dcr\_exact\_match\_rate} $= 0$), and MIA ROC-AUC $\approx 0.497$ (indistinguishable from chance at three seeds). No configuration shows gross memorisation at this cell, and the privacy picture is effectively a tie across HS3F, ForestFlow, and CTGAN; the compute-cost picture is separately covered in the paragraph above.

\paragraph{Why HS3F is the primary backbone for the main grid.}
Taken together, the matched-cell evidence at $(n_t{=}5, n_{\text{noise}}{=}5)$ is what motivates using HS3F as the primary backbone for the main hyperparameter sweep:
\begin{itemize}\tightlist
    \item \textbf{Temporal coherence.} Two of the four trajectory-level metrics (autocorrelation distance, cross-feature drift) favour HS3F by roughly $2\times$ over ForestFlow; the other two are tied or within seed noise.
    \item \textbf{Ranking utility.} HS3F's mortality ROC-AUC ($0.672$) is the highest of the three; under three seeds the HS3F--ForestFlow gap is not statistically significant, but the HS3F--CTGAN gap is. LOS macro-F1 is tied with ForestFlow.
    \item \textbf{Architectural alignment to the feature schema.} HS3F's explicit categorical heads are designed for a mixed continuous/binary/categorical feature set (binary intervention flags, categorical admission codes). ForestFlow's all-continuous flow requires post-hoc clipping of discrete features, and the measured $2\times$ increase in autocorrelation distance is consistent with this round-trip introducing hour-to-hour transition noise.
    \item \textbf{Computational cost.} HS3F trains $\approx 32\%$ faster than ForestFlow and $\approx 3.7\times$ faster than CTGAN at this cell. Across a $40$-cell grid this compounds into a decisive sweep-feasibility advantage.
    \item \textbf{Privacy parity.} Proximity (DCR) and membership-inference (DOMIAS-style) diagnostics are indistinguishable across backbones, so HS3F's utility and cost advantages are not bought at a measured privacy cost.
\end{itemize}

\paragraph{Caveats on the backbone decision.}
The backbone-axis evidence rests on a single matched $(n_t, n_{\text{noise}})$ cell with three trajectory seeds. The HS3F main grid is in progress; the ForestFlow and CTGAN full grids are not run. The HS3F--ForestFlow ROC-AUC gap at this cell is not statistically significant, and the backbone ordering on other cells may differ. Main-grid conclusions below (\S\textbf{Empirical Outcomes from the Hyperparameter Sweep}) are therefore HS3F-specific by construction; claims transfer to ForestFlow or CTGAN only under the matched-cell comparison. Extending the three-seed matched comparison to the full grid is flagged in \code{STATUS\_TRACKER.md}; the CSV-level artefacts (\code{results/backbone\_comparison/sweep\_results\_\{hs3f,forest\_flow\_copy,ctgan\}.csv}) already carry the full schema and are ready for the next regeneration pass.

\subsubsection{Trajectory-Level Coherence Metrics}
\label{sec:trajectory-coherence}

The trajectory-level metrics defined in the Evaluation chapter (autocorrelation distance, stay-length KS, transition smoothness ratio, temporal cross-feature correlation drift) are implemented in \code{src/synth_gen/evaluation/trajectory\_metrics.py} and integrated into \code{run\_sweep.py}, with dedicated unit coverage in \code{src/synth_gen/evaluation/tests/test\_trajectory\_metrics.py}. Matched-cell numerical values for all three backbones at $(n_t{=}5, n_{\text{noise}}{=}5)$ appear in Table~\ref{tab:backbone-trajectory}. The main grid \code{results/sweep\_results.csv} predates the trajectory-column additions, so a full-grid sweep of these metrics is deferred to the next canonical regeneration (tracked in \code{STATUS\_TRACKER.md} under ``Matched backbone comparison'' and ``Statistical reliability''). Implementation correctness is independently established by the unit tests, which verify:
\begin{itemize}\tightlist
    \item zero values when real and synthetic trajectories are identical (sanity bound);
    \item monotonic degradation as synthetic trajectories are perturbed with progressively larger noise;
    \item stable behaviour when trajectory lengths fall below the minimum window (short trajectories are excluded rather than causing NaNs).
\end{itemize}
Full-grid numerical outcomes will appear in the final thesis results table once regeneration completes; the present chapter already documents the design, integration, and matched-cell numerical output of these metrics, so the claim ``temporal coherence is evaluated'' is supported at both the implementation and the matched-cell empirical level.

\paragraph{Summary and cautious interpretation.}
The expanded sweep reveals clear trends that the pilot was too small to surface:
(1) $n_t$ is the dominant hyperparameter for utility and quality, with diminishing returns beyond $\approx 5$--$7$;
(2) $n_{\text{noise}}$ has a weaker and more configuration-dependent effect;
(3) the accuracy--F1 divergence at high $n_t$ confirms minority-class collapse as the primary failure mode;
(4) range violations improve substantially at higher $n_t$ (from $\sim$30\% to $\sim$14\%);
(5) the remaining utility gap appears structural rather than hyperparameter-limited.
These main-grid results are single-backbone (HS3F) and single-dataset, so they support trend identification but not definitive performance claims; the three-seed matched-cell backbone comparison in \S\ref{sec:backbone-comparison} partially addresses the backbone-axis limitation, but a full-grid matched sweep remains outstanding.

\section{Limitations and Threats to Validity}
%==============================================================================

Empirical software-engineering guidance groups threats into construct, internal, external, and conclusion validity \cite{runesonHost2009CaseStudyGuidelines}.
The items below are stated directly: they bound what the current implementation and sweep can claim.

\subsection{Construct validity}

\paragraph{Operationalization of ``utility'' and ``fidelity''.}
Multi-task TSTR (logistic baselines in \code{tstr\_framework}) and the sweep's marginal/correlation/range metrics are \emph{proxies} for clinical usefulness and distributional faithfulness.
They still omit temporal consistency across full stays, calibration of predicted risks, and expert review of clinically implausible joint patterns.
\textbf{Mitigations in code:} two complementary patient-level tasks (mortality, LOS) reduce single-score overfitting; KS, $\|\Delta C\|_F$, and range violations deliberately stress different failure modes; post-decode clipping is intended to align known flags with binary semantics (\S~\textbf{Post-Decode Clinical Normalization}); and privacy is now monitored with DCR-based diagnostics plus DOMIAS-style MIA scoring (\S~\textbf{Privacy Risk Metrics}).
\textbf{Gaps:} no multivariate two-sample tests, no clinician validation loop, and no full adaptive/shadow-model MIA battery beyond the nearest-neighbor DOMIAS-style attack proxy. Additionally, binary intervention flags (\code{vent}, \code{vaso}) are transported as continuous flows during HS3F generation and rounded post-hoc; the fraction of generated values that fell outside $\{0,1\}$ before clipping is not currently tracked, leaving one source of KS inflation and plausibility degradation unmeasured.

\paragraph{Label and prevalence semantics.}
Encoded categories and rare events may dominate apparent TSTR scores (e.g., accuracy under imbalance).
\textbf{Mitigations:} macro-F1 and ROC-AUC on LOS; explicit discussion of F1 collapse in the sweep results (Figures~\ref{fig:sweep-f1-heatmap}--\ref{fig:sweep-f1-lines}).
\textbf{Gaps:} prevalence-aware baselines and stratified reporting are not yet first-class in the sweep output.

\subsection{Internal validity}

\paragraph{Confounding by configuration and compute.}
The sweep varies both $n_t$ and $n_{\text{noise}}$, but wall-clock differences (e.g., training time varies from $\approx 5$ minutes to over three hours) can couple ``better'' metrics with unequal optimisation budgets unless training is stopped by a common criterion.
\textbf{Mitigations:} per-stage timings and degeneracy annotation are logged; interpretation already warns against ranking on this CSV alone.
\textbf{Gaps:} no matched early-stopping or compute-matched comparison protocol in the sweep.

\paragraph{Determinism versus accidental reuse.}
Seeds and artifact paths support repeat runs, but the transformed-data cache (\code{\_build\_cache\_signature}, \code{\_load\_transformed\_cache}) can silently reuse large arrays when signatures match---correct when inputs are unchanged, hazardous if a user assumes a fresh transform after a partial edit that does not bump mtime or path.
\textbf{Mitigations:} signature includes path, size, mtime, column names, and index vectors; cache miss overwrites.
\textbf{Gaps:} \code{run\_sweep.py} exposes \code{--refresh-transformed-cache}, but cache-refresh decisions are still manual; semantic changes that preserve file metadata can be missed without explicit operator checks.

\paragraph{Evaluation leakage and split stability.}
TSTR and quality metrics depend on how real/synthetic splits are formed; any overlap between training generator inputs and evaluation cohorts would inflate utility.
\textbf{Mitigations:} the chapter ties evaluation to the implemented split conventions in \code{tstr\_framework} and preprocessor indices.
\textbf{Gaps:} formal independence guarantees across hour-0, autoregressive training, and TSTR splits should be audited line-by-line for each experiment script version.

\subsection{External validity}

\paragraph{Single database and pipeline.}
Results apply to MIMIC-derived hour-0 plus autoregressive ICU rows, the chosen feature set, and the Forest-Flow/HS3F backbones implemented in this work---not to ward workflows, other charting systems, or primary-care EHRs.
\textbf{Mitigations:} two-stage design and backbone selection document the scope of the generative family.
\textbf{Gaps:} no cross-cohort or cross-hospital replication; preprocessor and item definitions remain cohort-specific.

\paragraph{Synthetic sample size and conditioning.}
Conditional generation reuses real conditioning rows in sweep mode; this anchors context but limits exploration of counterfactual trajectories and may hide defects in the conditional model when the condition distribution is narrow.
\textbf{Gaps:} ablations over synthetic cohort size and purely synthetic conditioning chains are future work.

\subsection{Conclusion validity}

\paragraph{Statistical power and multiplicity.}
The expanded sweep ($\approx 40$ configurations) now repeats each cell across three trajectory seeds and records mean, sample standard deviation, and $95\%$ confidence-interval half-widths for key utility, quality, and privacy metrics.
Scanning several metrics and tasks still raises multiplicity concerns, and with only three seeds per cell the intervals should be treated as uncertainty indicators rather than definitive significance tests.
\textbf{Mitigations:} resumable \code{sweep\_results.csv}, failure rows preserved, uncertainty columns exported, and overlap-aware interpretation rather than deterministic point-estimate ranking.
\textbf{Gaps:} larger seed counts, bootstrap CIs, and multiple-comparison discipline are still future work in \code{run\_sweep.py}.

\paragraph{Threshold-based degeneracy.}
\code{degenerate\_flag} encodes a fixed heuristic on near-random utility; it reduces false confidence but can miss structured failure (good average AUROC with pathological calibration) or flag borderline runs arbitrarily.
\textbf{Mitigations:} flagged rows are retained, not dropped.
\textbf{Gaps:} sensitivity analysis for thresholds and correlation with distributional metrics is outstanding.

\section{Unified Sweep Architecture}
%==============================================================================

\subsection{Per-Configuration Lifecycle}

For each $(n_t, n_{\text{noise}})$ pair, \code{run\_sweep.py} executes:
\begin{enumerate}\tightlist
    \item Train hour-0 IID model,
    \item Train autoregressive model,
    \item Generate synthetic samples,
    \item Run multi-task TSTR + quality metrics,
    \item Append one normalized CSV row.
\end{enumerate}

Each finished configuration is normalized to the canonical header list \code{SWEEP\_RESULT\_COLUMNS} before append, so partial reruns do not corrupt \code{results/sweep\_results.csv}.
The row payload includes utility metrics, quality metrics, timing fields for both model stages, and \code{degenerate\_flag}.

\subsection{Always-On Transformed Cache}

Autoregressive transformed arrays are cached by default under \code{models/sweep\_cache/transformed\_autoregressive/} and reloaded when a signature check passes.

The signature dict in \code{metadata.pkl} is produced by \code{\_build\_cache\_signature} in \code{run\_sweep.py}: CSV path, size, and nanosecond mtime; \code{n\_cols} and \code{n\_target}; SHA-256 hashes of the ordered column-name list, \code{target\_indices}, and \code{condition\_indices}.
\code{\_load\_transformed\_cache} requires exact equality of all signature fields before reusing cached \code{.npy} arrays.

A hit skips redundant \code{transform} work; a miss rebuilds matrices and overwrites the cache, preventing reuse after the input file, schema, or index split changes.

\subsection{Resumability and Failure Logging}

Runs are keyed by \code{(nt, n\_noise)} from \code{results/sweep\_results.csv}; any existing row for a key is skipped on rerun, including rows previously written after failures.
Failures do not abort the full sweep; they are written as explicit NaN/error rows, preserving an audit trail.

\subsection{Degeneracy Guardrail}

A run is flagged as degenerate when both patient-level synthetic-side thresholds are simultaneously met:
\code{mortality\_synth\_roc\_auc <= 0.55} and \code{los\_synth\_macro\_f1 <= 0.15}, with finite-value checks before thresholding.
Vasopressor metrics are no longer part of the degeneracy check following the TSTR framework revision described in the Evaluation chapter.
Importantly, runs are \emph{not} dropped; metrics are preserved and annotated with \code{degenerate\_flag=1}.

\section{Reproducibility and Reporting Conventions}
%==============================================================================

The current report and code follow reproducibility-oriented implementation practices:
\begin{itemize}\tightlist
    \item deterministic seeds for NumPy/model constructors,
    \item explicit artifact paths and metadata-rich pickles,
    \item canonical CSV schema enforcement before append,
    \item transparent warnings instead of silent failures,
    \item progress instrumentation for long-running stages.
\end{itemize}

These choices are consistent with reproducible computational research practice and are intended to support debugging and thesis traceability.

\section{Relationship to External Methods and Projects}
%==============================================================================

Our implementation is grounded in:
\begin{itemize}\tightlist
    \item flow matching for CNF-style training \cite{lipman2023flow},
    \item tree-based flow matching for tabular generation \cite{jolicoeur2023forest},
    \item heterogeneous sequential mixed-type generation \cite{akazan2024hs3f},
    \item XGBoost as the core function approximator \cite{chen2016xgboost}.
\end{itemize}

Common synthetic tabular/time-series baselines include CTGAN \cite{xu2019ctgan}, TimeGAN \cite{yoon2019timegan}, TabDDPM \cite{kotelnikov2023tabddpm}, TabSyn \cite{zhang2024tabsyn}, and SDV ecosystem tooling \cite{sdvproject}. In this project, CTGAN/CRGAN is integrated through \code{src/synth_gen/models/ctgan\_adapter.py} as a pragmatic lower-bound/reference baseline inside the same data and metric pipeline; numerical results from the matched three-seed comparison at $(n_t{=}5, n_{\text{noise}}{=}5)$ appear in \S\ref{sec:backbone-comparison}. However, CTGAN is not implemented as a fully autoregressive per-row conditioned generator: adding that behaviour would require an architecture change (new sequential conditioning/generation design), which is outside the thesis scope. Transformer-based generators such as GReaT \cite{borisov2023great} and REaLTabFormer \cite{solatorio2023realtabformer} remain future comparison points.
