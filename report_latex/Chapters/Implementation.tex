\section{Operational Pipeline for Hyperparameter Sweep}
%==============================================================================

The sweep workflow is organised into four stages: data preparation, two-stage model training (hour-0 IID plus autoregressive transition), synthetic generation, and integrated evaluation with schema-controlled logging. The architecture is backbone-selectable (\code{forest-flow} or \code{hs3f}), and each configuration is recorded as one auditable experiment row so utility, fidelity, privacy, and runtime remain directly comparable.

Script-level inventories and command recipes are intentionally maintained in repository documentation and appendix material rather than repeated in this chapter. This section therefore focuses on implementation structure and evidential traceability rather than operational command detail.

\subsection{Feature Selection Policy}
\label{sec:feature-selection-policy}

Feature selection in this project is implemented as a deterministic preprocessing policy rather than as a wrapper method (for example recursive elimination) or an embedded model-importance threshold.
The objective is to keep clinically meaningful coverage while removing columns that are non-informative, structurally unsuitable for generation, or unstable for the autoregressive interface.

For MIMIC autoregressive preparation (\code{scripts/mimic/prepare\_autoregressive.py}), the selection steps are:
\begin{enumerate}\tightlist
    \item Remove columns that are 100\% missing across the source table (identified by a full-file DuckDB scan, cached for reproducibility).
    \item Remove datetime/timestamp fields (\code{DEFAULT\_DATETIME\_COLUMNS}) that encode logging context rather than physiological state.
    \item Apply the shared pruning policy (\code{columns\_to\_drop\_default\_feature\_pruning}), which removes known low-value or high-sparsity inputs (including selected biophysical \code{*\_std}/\code{*\_count} derivatives) before autoregressive expansion.
    \item Partition the retained variables into static and dynamic groups, then construct lag-1 autoregressive rows so current-time dynamic variables become generation targets and static + lagged variables become conditioning features.
\end{enumerate}

The fitted preprocessor (\code{scripts/mimic/fit\_autoregressive\_preprocessor.py}) then preserves this selected feature set, assigns feature types (numeric, binary, categorical), and stores explicit transformed-space indices for target and condition blocks.
This makes feature inclusion and model inputs reproducible across runs and explains why later sweep comparisons are interpretable as model/backbone effects rather than ad-hoc column-selection differences.

\section{Training Implementation Details}
%==============================================================================

\subsection{Hour-0 IID Training}

\code{train\_hour0.py} loads \code{models/hour0\_preprocessor.pkl}, transforms all hour-0 rows, and fits an IID generator:
\begin{equation}
\hat{p}(\mathbf{z}_0), \quad \mathbf{z}_0=[\mathbf{s},\mathbf{x}_0].
\end{equation}

The script persists model metadata including \code{model\_type}, training hyperparameters, and preprocessor linkage.
This stage exists to model initial-state realism separately from transition dynamics, which explains later findings where rollout quality and hour-0 plausibility diverge.

\subsection{Autoregressive Training}

\code{train\_autoregressive.py} loads \code{models/preprocessor\_full.pkl}, transforms selected rows of \code{autoregressive\_data.csv}, and reconstructs split matrices via persisted indices:
\begin{equation}
X_{\text{target}} = X[:, \text{target\_indices}], \quad
X_{\text{condition}} = X[:, \text{condition\_indices}].
\end{equation}

This removes the previous dependence on naive prefix slicing and is needed in the current implementation when transformed columns are type-grouped.
The design choice prevents target/condition leakage from column-order assumptions, which directly supports the validity of the reported TSTR and fidelity metrics.

\subsection{Categorical Typing Interface}

Both backbones receive \code{feature\_types} from preprocessor metadata:
\begin{itemize}\tightlist
    \item \code{'q'} quantitative,
    \item \code{'c'} categorical.
\end{itemize}

Forest-Flow internally overrides noised target dimensions to quantitative mode (flow arithmetic constraint), while HS3F keeps explicit categorical heads \cite{akazan2024hs3f}.
This was implemented to keep both backbones compatible with one data interface while preserving HS3F's mixed-type path, which is why later backbone differences are interpretable as modelling effects rather than pipeline mismatch.

\subsection{Categorical Labels and Flow Integration (Codebase)}

For HS3F categorical dimensions, training remaps observed categories to contiguous local class indices for XGBoost multiclass fitting; \code{sample()} inverts that map via stored \code{local\_to\_orig} so synthetic rows use the same encoded IDs as the preprocessor artifact.
The training remap yields contiguous class indexing under arbitrary subsampling without sparse multiclass labels at fit time.

For continuous targets under Forest-Flow, sampling follows explicit first-order integration.
Forest-Flow integrates the learned velocity field from noise to data with explicit forward-Euler updates (\code{X\_t += h * Y\_hat}) in \code{src/lexisflow/models/forest\_flow.py}; this is the standard conditional flow-matching integration path, not the backward (score-based) diffusion formulation.
HS3F continuous dimensions default to \code{solver="rk4"} in \code{src/lexisflow/models/hs3f.py}, with \code{solver="euler"} available as an alternative.

\section{Sampling and Reconstruction}
%==============================================================================

\subsection{Model-Space Sampling}

For conditional generation, models sample:
\begin{equation}
\tilde{X}_{\text{target}} \sim p_{\theta}(X_{\text{target}} \mid X_{\text{condition}}).
\end{equation}

In sweep mode, conditioning rows are sampled from the real transformed condition matrix and reused as context for synthetic targets.

\subsection{Rebuilding Full Feature Vectors}

The implementation reconstructs full transformed rows in preprocessor order:
\begin{equation}
X_{\text{full}}[:,\text{target\_indices}] \leftarrow \tilde{X}_{\text{target}}, \quad
X_{\text{full}}[:,\text{condition\_indices}] \leftarrow X_{\text{condition}}.
\end{equation}

Then \code{inverse\_transform()} restores tabular values and category labels.
This reconstruction step is required so all downstream evaluators operate in comparable tabular semantics rather than backbone-specific latent coordinates.

\subsection{Post-Decode Clinical Normalisation}

Known intervention indicators (e.g., \code{vent}, \code{vaso}) are explicitly clipped/rounded to $\{0,1\}$ before evaluation/export.
This step is intended to restore discrete $\{0,1\}$ semantics for binary treatment flags after decode.

\section{Evaluation Stack}
%==============================================================================

\subsection{Cohort-Level Data Splits}
\label{sec:cohort-splits}

All utility, fidelity, and privacy metrics are computed against real-data cohorts that are patient-disjoint from the generator's training cohort.
At preprocessing time (\code{scripts/prepare\_autoregressive.py}), the autoregressive table is partitioned at the \code{subject\_id} level into three CSVs with an 80\,/\,10\,/\,10 split drawn from a fixed RNG seed so the partition is reproducible across runs:
\begin{itemize}\tightlist
    \item \code{data/processed/autoregressive\_data.csv} -- the training cohort (80\,\% of patients), the only file the generator ever sees.
    \item \code{data/processed/real\_test.csv} -- a TSTR / fidelity reference cohort (10\,\% of patients).
    \item \code{data/processed/real\_holdout.csv} -- a privacy-evaluation holdout cohort (10\,\% of patients).
\end{itemize}

Because the split is performed on \code{subject\_id}, no patient's timesteps ever span the training cohort and an evaluation cohort.
Consequently:
\begin{itemize}\tightlist
    \item Multi-task TSTR (section \textbf{Multi-Task TSTR}) trains classifiers on synthetic data and tests them on sequences drawn from \code{real\_test.csv}; the real-trained baseline classifier is also fit and evaluated entirely within this disjoint cohort via an internal stratified split (\code{test\_size=0.3}).
    \item The distributional fidelity metrics (KS, correlation Frobenius norm, range-violation rate) are computed against a sample of \code{real\_test.csv} rather than the training matrix, so they measure distributional recovery of \emph{unseen} rows rather than reproduction of the training set.
    \item The DCR overfitting-protection and DOMIAS-style membership-inference attack use \code{real\_holdout.csv} directly as the holdout reference, giving the overfitting-vs-generalisation comparison a meaningful signal even when the generator is expressive enough to memorise.
\end{itemize}

The split is materialised once per preprocessing run and referenced explicitly by \code{AutoregressiveInputs.real\_test\_path} and \code{AutoregressiveInputs.real\_holdout\_df} in \code{src/lexisflow/sweep/data\_prep.py}, ensuring the sweep driver, the matched backbone-comparison driver, and any future evaluation scripts consume byte-identical reference cohorts.

\subsection{Multi-Task TSTR}

The active sweep uses a unified patient-level TSTR framework with two tasks:
\begin{enumerate}\tightlist
    \item Mortality classification (binary, patient-level),
    \item ICU LOS category classification (multi-class, patient-level).
\end{enumerate}

Each task uses a \code{\_PaddedSequenceClassifier}: patient trajectories are grouped by \code{subject\_id}, sorted by \code{hours\_in}, padded to a common length with a binary mask, flattened, and fed to a logistic-regression baseline. This patient-level design eliminates within-patient data leakage and captures the temporal structure the autoregressive generator is designed to preserve. The design evolution from the original row-level three-task framework and the rationale for removing vasopressor-use prediction are documented in the Evaluation chapter. The TSTR paradigm follows established synthetic health-data evaluation practice \cite{esteban2018rcgan}.

\subsection{Task Metrics}

For binary tasks, reported metrics include accuracy, F1, and ROC-AUC.
For LOS (multi-class), macro-F1 and one-vs-rest macro ROC-AUC are included.

\subsection{Distributional Quality Metrics}

In addition to task utility, sweep evaluation computes:
\begin{enumerate}\tightlist
    \item Average KS statistic across valid numeric columns,
    \item Correlation matrix Frobenius norm,
    \item Clinical range violation percentage.
\end{enumerate}

Let $\mathcal{J}$ be valid numeric columns:
\begin{equation}
\text{KS}_{\text{avg}} = \frac{1}{|\mathcal{J}|}\sum_{j\in\mathcal{J}}\sup_x |F_j^{\text{real}}(x)-F_j^{\text{synth}}(x)|.
\end{equation}
\begin{equation}
\text{CorrDiff} = \|C^{\text{real}} - C^{\text{synth}}\|_F.
\end{equation}

Columns with insufficient finite samples or zero variance are excluded before these computations to avoid undefined correlations.

\subsection{Privacy Risk Metrics}

To complement utility/fidelity metrics, the current implementation adds four privacy checks used in recent tabular-synthetic evaluation practice \cite{lautrup2024syntheval,breugel2023domias}:
\begin{enumerate}\tightlist
    \item Distance to Closest Record (DCR) summary ($\mathrm{median}$, tails, exact-match rate),
    \item DCR baseline-protection score (synthetic-vs-random closeness to real data),
    \item DCR overfitting-protection score (synthetic closeness to train vs holdout),
    \item DOMIAS-inspired membership inference attack (MIA) ROC-AUC, average precision, and attacker advantage.
\end{enumerate}

For synthetic samples $\tilde{x}_i$ and real training set $\mathcal{D}_{\text{train}}$, DCR is:
\begin{equation}
\mathrm{DCR}(\tilde{x}_i,\mathcal{D}_{\text{train}})=\min_{x\in\mathcal{D}_{\text{train}}}\|\phi(\tilde{x}_i)-\phi(x)\|_2,
\end{equation}
with mixed-type embedding $\phi(\cdot)$ (min-max normalised numerics + one-hot categoricals).
Baseline protection follows the SDMetrics ratio form \cite{sdmetricsPrivacyDocs}:
\begin{equation}
\mathrm{Score}_{\text{baseline}}=\min\left(1,\frac{\mathrm{median}\,\mathrm{DCR}_{\text{synth}\rightarrow\text{train}}}{\mathrm{median}\,\mathrm{DCR}_{\text{random}\rightarrow\text{train}}}\right).
\end{equation}

Overfitting protection compares train-vs-holdout nearest-neighbour proximity for synthetic rows, where the holdout is the patient-disjoint \code{real\_holdout.csv} cohort defined in section~\ref{sec:cohort-splits}:
\begin{equation}
\mathrm{Score}_{\text{overfit}}=\min\left(1,\,2\left(1-\Pr\left[\mathrm{DCR}_{\text{synth}\rightarrow\text{train}} < \mathrm{DCR}_{\text{synth}\rightarrow\text{holdout}}\right]\right)\right).
\end{equation}

Membership risk is quantified with a DOMIAS-style local overfitting score \cite{breugel2023domias}:
\begin{equation}
s(x)=\log\frac{d_{\text{ref}}(x)+\epsilon}{d_{\text{synth}}(x)+\epsilon},
\end{equation}
where $d_{\text{synth}}$ is nearest-neighbour distance to synthetic data and $d_{\text{ref}}$ is distance to the patient-disjoint holdout cohort, so a non-zero attacker advantage can only reflect closeness to training patients rather than to the evaluation cohort itself.
Attack performance is reported via ROC-AUC (and attacker advantage) in the membership-inference paradigm \cite{shokri2017membership}.

\paragraph{Interpretation guardrail.}
DCR-style proximity metrics are useful diagnostics but should not be treated as sufficient legal/anonymity evidence on their own; recent work shows distance-only proxies can miss true membership leakage, so MIA results remain primary for privacy claims \cite{yao2025dcrdelusion}.

\subsection{Empirical Outcomes from the Hyperparameter Sweep}

One normalised row per completed $(n_t, n_{\text{noise}})$ cell is written to the dataset-specific sweep CSV (MIMIC: \code{results/sweep\_results.csv}). Quantitative interpretation of those columns is in Chapter~7; column semantics and analysis scripts are covered in the User Guide and Source Code appendices.

\paragraph{Trajectory-level coherence}
\label{sec:trajectory-coherence}

The trajectory metrics used in Chapter~7 (autocorrelation distance, stay-length KS, smoothness, temporal drift) are implemented in \code{evaluation/trajectory\_metrics.py} and called from the sweep driver. Numerical results and discussion are in Chapter~7; unit tests in the repository check identity, monotonicity under noise, and short-trajectory edge cases.

\section{Sweep driver, cache, and logging}
\label{sec:unified-sweep}
%==============================================================================

For each $(n_t, n_{\text{noise}})$ cell, the driver trains hour-0 and autoregressive models, generates a synthetic cohort, and runs the evaluation stack, appending one normalised row to the sweep CSV. Headers are fixed so partial reruns cannot corrupt the log. Transformed autoregressive arrays may be cached when metadata matches, reducing repeated work across the grid. Completed keys are skipped on rerun; failed cells are still recorded (with error or NaN fields) so the grid has a full audit trail. CLI entrypoints, profile names, cache paths, and worker flags are not duplicated here: they are documented in the User Guide (companion appendix).

\paragraph{Degeneracy flag.}
A cell is marked \code{degenerate\_flag=1} when \emph{both} \code{mortality\_synth\_roc\_auc <= 0.55} and \code{los\_synth\_macro\_f1 <= 0.15} (with finite checks), reflecting catastrophically poor synthetic-side utility. Rows are not dropped, so Chapter~7 can report failed and healthy cells side by side.

\section{Cross-Dataset Reproducibility on a Public Benchmark}
\label{sec:challenge2012-impl}
%==============================================================================

Access to MIMIC-III requires credentialed PhysioNet approval, which is a barrier to external reproduction of the main results. To make the pipeline reproducible for a reader without that credential, the implementation additionally targets the PhysioNet Challenge 2012 set-A cohort \cite{silva2012challenge}, an open-access ICU benchmark released under the Open Data Commons Attribution licence (ODC-BY) and available at \url{https://physionet.org/content/challenge-2012/1.0.0/}. The Challenge 2012 path runs on 4{,}000 ICU patients gridded to 48 hourly bins, yielding $189{,}120$ rows after the same static/dynamic separation used for MIMIC.

\subsection{What Data These Cohorts Contain}

Although both datasets are ICU-focused, they differ in source breadth. MIMIC-III is a full hospital critical-care EHR with broad modality coverage (demographics, charted vital signs, laboratory results, interventions, medications, procedures, and outcomes). In this project, the modelling interface uses the structured hourly tabular slice defined by the preprocessing contract in section~\ref{sec:mimic-preprocess-contract}, so each synthetic trajectory is generated from static patient descriptors plus time-indexed physiological and intervention features.

PhysioNet Challenge 2012 set-A is a narrower benchmark specifically packaged for mortality prediction from the first 48 ICU hours. In this pipeline it is prepared into the same autoregressive schema as MIMIC: static demographics, hourly aggregated clinical measurements, and the two downstream labels used throughout this report (\code{hospital\_expire\_flag} and \code{los\_icu}). This alignment is intentional, because it allows the exact same training and evaluation code paths to run on an open dataset.

\subsection{Shared pipeline, dataset-specific preparation}

Reproducibility here means the same code paths, not a second implementation. After preparation, Challenge 2012 and MIMIC share preprocessor fitting, training, sampling, and the full metric stack; only \code{scripts/challenge2012/} versus \code{scripts/mimic/} preparation differs. How to invoke the driver for each dataset is in the User Guide (companion appendix).

\subsection{Feature-Schema Alignment}

Challenge 2012 is semantically aligned with the MIMIC autoregressive schema: static demographics, vital-sign medians per hour, and the same mortality / ICU length-of-stay labels (\code{hospital\_expire\_flag}, \code{los\_icu}). Two columns with below-10\,\% coverage (\code{TroponinI}, \code{Cholesterol}) are dropped during preparation, and the SAPS-I / SOFA severity scores are not used, so the downstream feature vector slots into the same preprocessor contract as the MIMIC cohort.

\paragraph{Challenge-specific utility-oriented denoising.}
During Challenge 2012 debugging, near-random TSTR utility was traced to a mismatch between ``distributional realism'' and ``predictive transfer'': some variables showed weak outcome signal in the real cohort while also exhibiting large real--synthetic drift, which increased model noise without improving downstream discrimination. To address this at the \emph{pipeline} level (rather than by post-hoc evaluator filtering), the Challenge preparation and preprocessor scripts now remove an additional fixed set of columns:
\begin{itemize}\tightlist
    \item \code{Temp}, \code{PaO2}, \code{PaCO2}, \code{ALP},
    \item \code{SaO2}, \code{Glucose}, \code{Platelets}, \code{Height}.
\end{itemize}
This policy is deterministic and encoded directly in \code{scripts/challenge2012/prepare\_hour0.py}, \code{prepare\_autoregressive.py}, \code{fit\_hour0\_preprocessor.py}, and \code{fit\_autoregressive\_preprocessor.py}. Consequently, generation, sampling, and evaluation all use the same reduced feature schema, avoiding the methodological inconsistency of training on one schema and evaluating on another.

\paragraph{Interpretation and risk.}
This is a conservative denoising step, not a claim that the removed variables are universally non-clinical. The practical thesis objective is to improve utility under a fixed model family and finite compute budget; therefore columns are removed only when evidence suggests they contribute more noise/drift than transferable signal in this specific pipeline. Residual risk remains that some dropped variables could become useful under different backbones or richer temporal models, so this policy should be revisited if the modelling architecture changes.


\subsection{Status and Role in This Thesis}

The Challenge 2012 sweep is in progress at the time of writing; its results are intentionally out of scope for the empirical claims of this dissertation. In this thesis, Challenge 2012 is used strictly as a reproducibility supplement: it demonstrates that the same pipeline can be re-executed on an open dataset by readers without MIMIC credentialing. The evidential core of the dissertation remains the MIMIC-III analysis reported in Chapter~7. Numerical Challenge 2012 outcomes are deferred to follow-on work after sweep completion.
