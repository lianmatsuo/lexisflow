\chapter{Requirements and Specification}

This chapter states what the implementation must do and how that is verified in experiments. The lists exist for \emph{traceability}: they connect the research objectives in Chapter~1 to concrete obligations on the code and to the metrics produced by the evaluation and sweep drivers. They should be read as technical requirements for a credentialed \emph{research} workflow, not as a product specification, a governance case, or a claim of fitness for external release of synthetic patient data. Priorities use MoSCoW (Must, Should, Could, Will not have this time) \cite{agileBusinessConsortiumMoSCoW}. The axes (fidelity, utility, privacy) follow synthetic health-data evaluation practice in the literature \cite{yan2022benchmarking,kaabachi2025scoping}; any informal adequacy bands used when interpreting results are stated in the Evaluation chapter and are research-stage, not operational thresholds.

%==============================================================================
\section{Research scope}
%==============================================================================

Work on synthetic EHRs is often discussed in terms of data holders, downstream modellers, and privacy review. Those roles explain \emph{why} fidelity, TSTR utility, and privacy diagnostics matter, but the dissertation's contribution is the integrated pipeline, models, and metrics---not a workflow analysis or release process. The sections below therefore omit actor--use-case narratives and state requirements directly. Data preparation follows the preprocessing contract in section~\ref{sec:mimic-preprocess-contract}; training, generation, and evaluation are batch jobs run in a research setting with the constraints listed under Known limitations.

\section{Specification}
%==============================================================================

\subsection{Data Specification}

\begin{itemize}\tightlist
    \item \textbf{Input}: MIMIC-III hourly ICU tables produced under the preprocessing contract in section~\ref{sec:mimic-preprocess-contract}, containing mixed-type features (continuous vital signs, binary intervention flags, categorical codes) and static demographics.
    \item \textbf{Autoregressive format}: Each row represents one patient-hour, with target columns (current-hour dynamic features) and condition columns (static covariates plus lagged dynamic features).
    \item \textbf{Hour-0 format}: One row per patient containing the joint static and initial dynamic state vector.
    \item \textbf{Output}: Synthetic CSV files in the same schema as the autoregressive input, decodable via the fitted preprocessor.
\end{itemize}

\subsection{Model Specification}

\begin{itemize}\tightlist
    \item \textbf{Backbone}: ForestFlow (all-continuous flow matching with XGBoost regressors) or HS3F (heterogeneous sequential generation with flow regressors for continuous and XGBoost classifiers for categorical features).
    \item \textbf{Time discretisation}: Configurable number of time levels $n_t$ (default: 20).
    \item \textbf{Noise multiplier}: Configurable $n_{\text{noise}}$ controlling training data augmentation (default: 50).
    \item \textbf{XGBoost parameters}: Number of estimators, max depth, and learning rate, all configurable via YAML.
    \item \textbf{Two-stage architecture}: IID hour-0 model plus conditional autoregressive model, trained independently but evaluated jointly in the sweep.
\end{itemize}

\subsection{Evaluation Specification}

\begin{itemize}\tightlist
    \item \textbf{Utility}: Multi-task TSTR with logistic regression baselines reporting accuracy, F1, and ROC-AUC per task.
    \item \textbf{Fidelity}: Per-feature KS statistics, correlation matrix Frobenius norm, clinical range violation percentage.
    \item \textbf{Privacy}: DCR median and match rate, baseline and overfitting protection scores, DOMIAS-style MIA ROC-AUC and attacker advantage.
    \item \textbf{Sweep}: Grid search over $(n_t, n_{\text{noise}})$ with results logged to a canonical CSV with schema enforcement.
\end{itemize}

\section{Known Limitations and Constraints}
%==============================================================================

\begin{itemize}\tightlist
    \item \textbf{Data access}: MIMIC-III requires credentialing and a signed data use agreement; the system cannot be tested without this access.
    \item \textbf{Computational cost}: Sweep runtime scales as $O(n_t \times n_{\text{noise}} \times d_x)$ per configuration, which at 8$\times$8 Fibonacci grid scale yields wall-clock costs between $\approx$3 minutes and $\approx$18.9 hours per cell; low-cost cells correspond to small $(n_t, n_{\text{noise}})$, high-cost cells to $(n_t{=}34, n_{\text{noise}}{=}34)$.
    \item \textbf{Single dataset}: All requirements are specified and verified against MIMIC-III; portability to other clinical datasets is not guaranteed.
    \item \textbf{No real-time or interactive generation}: The system operates in batch mode via command-line scripts.
\end{itemize}
