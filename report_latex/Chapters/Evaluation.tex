\section{Software Testing}

The evaluation stack is validated with deterministic \code{pytest} unit tests before computationally expensive sweeps are executed \cite{pytestProject}. Coverage focuses on directional correctness for quality and privacy metrics, defensive handling of edge cases (\code{NaN}s, low-sample columns, schema consistency), and report-shape stability. In particular, the tests verify expected behaviour under identity, shift, and memorisation-stress fixtures.

These checks are implementation-level guards rather than inferential evidence. Full numerical interpretation and validity claims are based on the sweep analysis reported in this chapter. Detailed test inventories and fixture-level assertions are provided in the appendix and repository test suite, so the chapter remains focused on empirical interpretation.

\section{Evaluation Methodology and Metric Rationale}
\label{sec:eval-methodology}
%==============================================================================

This section specifies the evaluation protocol used in this project. It focuses on methodological design and metric rationale, not numerical outcomes.

\subsection{Sweep-Oriented Evaluation Protocol}

Evaluation is operationalised as a reproducible sweep loop (\code{scripts/run\_sweep.py}) over $(n_t, n_{\text{noise}})$:
\begin{enumerate}\tightlist
    \item train hour-0 IID model and autoregressive model with the same hyperparameters;
    \item generate synthetic rows conditioned on sampled real contexts;
    \item evaluate utility (multi-task TSTR), statistical quality, and privacy;
    \item append one canonical row to \code{results/sweep\_results.csv}.
\end{enumerate}

The sweep is resumable by design (existing rows are skipped), logs failed runs with explicit error fields, and maintains schema compatibility through migration checks. To reduce recomputation, transformed matrices are cached with signatures over source-table metadata and split-index hashes.

\subsection{Baseline Scope and Interpretation}
\label{sec:baseline-scope}

The evaluation includes a CTGAN baseline as a \emph{reference lower bound}, not as a fully matched autoregressive competitor. In the current implementation, CTGAN is trained on the transformed joint table but sampled without per-row autoregressive conditioning. This means it does not consume the full row-specific context vector at generation time in the same way as the ForestFlow and HS3F transition models.

Making CTGAN fully autoregressive in this pipeline would require an architecture-level redesign (for example, a new sequential wrapper with explicit step-wise conditioning logic, state propagation, and compatibility checks across mixed continuous/categorical features), rather than a small hyperparameter or script-level modification. Delivering that redesign with adequate validation would expand the implementation and experiment surface beyond the scope of this thesis.

Accordingly, CTGAN results are interpreted conservatively: they provide a practical baseline reference and lower-bound comparator for utility/quality/privacy trends, but they are not used to make strong claims about parity with fully conditional autoregressive generators.

\subsection{Utility Evaluation via Multi-Task TSTR}

Primary utility assessment follows the Train-on-Synthetic, Test-on-Real (TSTR) paradigm, a widely used protocol in synthetic healthcare-data evaluation \cite{esteban2018rcgan,yan2022benchmarking}. The core idea is to train a downstream clinical predictor on synthetic data only and evaluate it on real held-out patients; the resulting performance gap relative to a real-trained baseline quantifies how much predictive signal the synthetic generator preserves.

\paragraph{Protocol decision and rationale.}

The initial row-level TSTR setup was replaced by a patient-level sequence protocol because row-level evaluation introduced structural bias: potential within-patient leakage, weak sensitivity to temporal coherence, and instability under sparse-event class imbalance. The revised protocol aligns the utility test with the intended use case of patient-level trajectory modelling.

Let $\mathcal{S}_i = \{\mathbf{x}_{i,0}, \ldots, \mathbf{x}_{i,T_i}\}$ denote the ordered hourly sequence for patient $i$, with patient-level label $y_i$. The synthetic-trained sequence classifier $g_{\phi}$ is evaluated as:

\begin{equation}
\phi_{\text{synth}} = \arg\min_{\phi}\mathcal{L}(g_{\phi}(\mathcal{S}_{\text{synth}}), y_{\text{synth}}), \quad
\text{evaluate } g_{\phi_{\text{synth}}} \text{ on } (\mathcal{S}_{\text{real,test}}, y_{\text{real,test}}).
\end{equation}

Patient-level splitting is enforced by \code{subject\_id}: the generator trains on one patient cohort, and utility/fidelity/privacy evaluation uses disjoint real cohorts. Within the utility cohort, the real-trained baseline uses an internal stratified patient split. Sequences are sorted, padded with masks, flattened, and evaluated with a class-balanced logistic baseline. This yields a leakage-resistant, temporally aware utility measure without introducing a new sequence-model confound.

The vasopressor task was removed from the final utility set because, in this cohort, it provided limited additional discrimination beyond mortality and LOS signals \cite{yan2022benchmarking}. The final framework therefore evaluates:

\begin{itemize}\tightlist
    \item mortality prediction (binary, patient-level),
    \item ICU length-of-stay category prediction (multi-class, patient-level).
\end{itemize}

Task models use class-balanced logistic-regression baselines from scikit-learn \cite{pedregosa2011scikit}. Reported metrics are accuracy, F1 (macro F1 for multi-class), and ROC-AUC (binary or one-vs-rest macro) \cite{fawcett2006roc}, with a real-trained reference model computed in parallel.

\subsection{Statistical Quality Metrics}

Complementary to TSTR, the project computes feature-space quality metrics in \code{quality\_metrics.py}:
\begin{enumerate}\tightlist
    \item \textbf{Average KS statistic} across common numeric features, via two-sample KS:
    \begin{equation}
    \overline{KS} = \frac{1}{|\mathcal{J}|}\sum_{j\in\mathcal{J}}\sup_x\left|F^{\text{real}}_j(x)-F^{\text{synth}}_j(x)\right|.
    \end{equation}
    \item \textbf{Correlation Frobenius norm}:
    \begin{equation}
    \Delta_{\text{corr}} = \left\|C_{\text{synth}} - C_{\text{real}}\right\|_{F}.
    \end{equation}
    \item \textbf{Clinical range violation percentage} over predefined physiologic bounds.
\end{enumerate}

This metric family aligns with synthetic-EHR benchmarking practice that combines distributional fidelity, relationship preservation, and plausibility checks \cite{yan2022benchmarking,kaabachi2025scoping}. The KS implementation relies on SciPy statistical tooling \cite{virtanen2020scipy}.

\subsection{Privacy Evaluation Metrics}

Privacy assessment in \code{privacy\_metrics.py} combines proximity diagnostics with attack-oriented indicators:
\begin{itemize}\tightlist
    \item DCR summary statistics (\code{median}, low-tail quantiles, exact-match rate),
    \item DCR baseline protection score (synthetic versus random baseline),
    \item DCR overfitting protection score (synthetic closeness to train versus holdout),
    \item DOMIAS-inspired membership inference metrics (ROC-AUC, average precision, attacker advantage).
\end{itemize}

The overfitting-protection and membership-inference metrics require a real reference cohort that the generator has never observed; both consume \code{real\_holdout.csv}, the 10\,\% patient-disjoint holdout carved at preprocessing time, passed into \code{compute\_privacy\_metrics(..., holdout\_df=...)} by the sweep driver. This disjointness is what makes the train-vs-holdout comparison in Score$_{\text{overfit}}$ and the attacker reference distance $d_{\text{ref}}$ in DOMIAS informative: a memorising generator cannot hide by being equally close to training and evaluation patients, because the two cohorts share no subjects.

The DCR-based protection scores follow SDMetrics-style formulations \cite{sdmetricsPrivacyDocs}. Membership inference follows the attack perspective introduced in classical MIA literature and modern synthetic-data attacks \cite{shokri2017membership,breugel2023domias}. Average precision is reported alongside ROC-AUC for ranking quality under class-imbalance sensitivity \cite{davisGoadrich2006}. Recent work also motivates caution: distance-only metrics should not be interpreted as standalone privacy guarantees \cite{yao2025dcrdelusion}.

\subsection{Trajectory-Level Quality Metrics}

The per-row metrics above assess marginal fidelity but cannot detect temporal incoherence within patient trajectories. A generator that produces correct marginal distributions may still fail to capture hour-to-hour dynamics, for example, yielding heart-rate sequences that jump erratically rather than evolving smoothly. To address this gap, the evaluation pipeline includes four trajectory-level metrics implemented in \code{trajectory\_metrics.py}:

\begin{enumerate}\tightlist
    \item \textbf{Autocorrelation distance} (\code{autocorr\_distance}): For each vital sign, the lag-1 autocorrelation is computed per trajectory and summarised by median. The metric reports the mean absolute difference in per-feature median autocorrelation between real and synthetic trajectories. A value of~$0$ indicates that synthetic trajectories preserve real temporal dependency; values above~$\sim 0.3$ indicate substantial structure loss.

    \item \textbf{Stay-length KS statistic} (\code{stay\_length\_ks}): A two-sample Kolmogorov--Smirnov test on the distribution of trajectory lengths (rows per \code{subject\_id}). Because the current generator produces fixed 48-hour trajectories while real ICU stays vary in length, this metric quantifies a known structural limitation of the fixed-window autoregressive scheme.

    \item \textbf{Transition smoothness ratio} (\code{transition\_mse\_ratio}): The ratio of mean hourly absolute change in each vital sign, synthetic versus real. A ratio of~$1.0$ means transitions are equally smooth; values above~$1$ indicate rougher (jumpier) synthetic trajectories; values below~$1$ indicate over-smoothed trajectories.

    \item \textbf{Temporal cross-feature correlation drift} (\code{temporal\_corr\_drift}): For clinically meaningful feature pairs (heart rate vs.\ MAP, systolic vs.\ diastolic BP, SpO\textsubscript{2} vs.\ respiratory rate), the within-trajectory Pearson correlation is computed per patient, and the metric reports the mean absolute difference in median correlation between real and synthetic data. This captures whether the generator preserves physiological relationships \emph{within} individual trajectories, not just in aggregate.
\end{enumerate}

These metrics are computed on trajectory-indexed data before per-row quality aggregation. Short trajectories are excluded where required for stable autocorrelation estimates. Uncertainty columns (\code{*\_std}, \code{*\_ci95}) are populated from multi-seed aggregation; in the current main grid, each cell is averaged over three seeds.

\subsection{Model-Screening Guardrail}

To reduce accidental over-interpretation of clearly weak runs, the sweep records a \code{degenerate\_flag} when the two patient-level synthetic-side utility metrics are simultaneously near-random: mortality ROC-AUC $\leq 0.55$ and LOS macro-F1 $\leq 0.15$. This is an experiment-management heuristic and is not used as a formal inferential criterion; flagged rows are retained in the sweep CSV rather than dropped, preserving an audit trail of degenerate configurations.

\subsection{Reporting Scope}

This chapter is the primary location for empirical interpretation. Chapter~6 documents implementation details; this chapter interprets utility, fidelity, privacy, and efficiency outcomes from \code{results/sweep\_results.csv} and related plot artefacts.

\section{Analysis of the Hyperparameter Sweep}
%==============================================================================

\subsection{Framing: Effect Characterisation Before Configuration Ranking}

This section is written as a response-surface analysis, not as a pure hyperparameter-tuning exercise. The primary objective is to characterise how changing $n_t$ and $n_{\text{noise}}$ affects utility, fidelity, privacy, temporal coherence, and compute cost, including whether effects are monotone, flat, or saturating. Reporting a ``best observed'' cell is secondary and is restricted to the populated region of the current grid. Extrapolation beyond that region is treated as indicative trend analysis only and is not used as a standalone empirical claim.

\subsection{Scope and Caveats}

The analysis below covers the Fibonacci-scale grid $n_t, n_{\text{noise}} \in \{1, 2, 3, 5, 8, 13, 21, 34\}$. In the canonical CSV snapshot used for this chapter, all $64$ of $64$ cells are complete. This removes the previous high-$n_t$/high-$n_{\text{noise}}$ corner uncertainty and allows direct interpretation of the full measured grid.

The main HS3F grid is aggregated over three trajectory seeds per cell. This reduces single-seed volatility, but with only three seeds the uncertainty bands should still be read as indicative rather than definitive.

A structural caveat applies throughout: the real-model TSTR baselines (mortality ROC-AUC $\approx 0.82$, LOS macro-F1 $\approx 0.77$) are computed on a patient-disjoint subset of \code{real\_test.csv} using the same logistic-regression sequence classifier as the synthetic-trained arm. They are therefore a tight upper bound for what a linear classifier can extract from this cohort, not an absolute ceiling, a non-linear classifier would raise the baseline further, which would in turn widen every gap reported below.

\subsection{Mortality TSTR: ROC-AUC, Accuracy, Balanced Accuracy, and F1}
\label{sec:mortality-tstr}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_heatmap.png}
\caption{Mortality TSTR ROC-AUC across the sweep grid (left) and the gap against the real-trained baseline (right).}
\label{fig:sweep_mort_roc}
\end{figure}

ROC-AUC is the most scientifically informative mortality metric in this regime because (i) the cohort is class-imbalanced and (ii) the synthetic-trained classifier frequently collapses to near-constant predictions, which inflates or deflates threshold-dependent metrics. Figure~\ref{fig:sweep_mort_roc} shows a clean monotone signal in $n_t$: marginal-mean ROC-AUC rises from $0.518$ at $n_t{=}1$ through $0.618$ at $n_t{=}13$ to $0.638$ at $n_t{=}34$. Across the $n_{\text{noise}}$ axis the trend is still flat (marginal means differ by only $\approx 0.026$ between the best and worst $n_{\text{noise}}$ bands, with no monotone pattern). The best observed cell mean is now $0.659$ at $(n_t{=}34, n_{\text{noise}}{=}21)$, while the smallest gap-to-real cell is $(n_t{=}21, n_{\text{noise}}{=}1)$ at $-0.165$. The gap-to-real panel mirrors the $n_t$ trend, narrowing from $\approx -0.35$ at the sparsest cell to $\approx -0.17$ at the best region, but never closing.

The interpretation is that $n_t$ controls the fidelity of the flow-matching integration path: with too few integration steps the target distribution is under-fit and predictors trained on the resulting samples carry almost no class signal. $n_{\text{noise}}$ controls the number of noise anchors per training row, which in principle should regularise the score-matching objective, but in this pipeline the downstream TSTR task is dominated by bias, not variance: adding noise anchors cannot rescue a mis-integrated velocity field. This is an architectural rather than a hyperparameter finding, and it implies that spending compute on larger $n_{\text{noise}}$ is poor value unless $n_t$ is already in the saturated region.

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_f1_heatmap.png}
\caption{Mortality TSTR F1 (left) and gap against the real-trained baseline (right).}
\label{fig:sweep_mort_f1}
\end{figure}

F1 on the mortality task (Figure~\ref{fig:sweep_mort_f1}) tells a more pessimistic story: values sit in $0.04$--$0.25$ against a real-model baseline of $\approx 0.47$--$0.49$, and the gap barely closes with $n_t$. This is consistent with a synthetic generator that preserves \emph{rank ordering} (which ROC-AUC measures) but systematically mis-calibrates the decision threshold. In other words, the synthetic mortality signal is directionally correct but the positive class is under-represented in the synthetic joint distribution, so the classifier's optimal operating point is pushed into a low-precision regime. The accuracy and balanced-accuracy heatmaps (not reproduced in full; \code{sweep\_accuracy\_heatmap.png} and \code{sweep\_balanced\_accuracy\_heatmap.png}) corroborate this: raw accuracy swings between $0.10$ and $0.85$ depending on which class the classifier defaults to, while balanced accuracy stays in $0.48$--$0.60$, just above chance. The only metric from this family worth reading for model selection is ROC-AUC; the others are structurally unstable under class-collapse behaviour.

\subsection{LOS TSTR: ROC-AUC, Macro-F1, Accuracy}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_los_macro_f1_heatmap.png}
\caption{LOS TSTR macro-F1 (left) and gap against the real-trained baseline (right).}
\label{fig:sweep_los_f1}
\end{figure}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_los_roc_heatmap.png}
\caption{LOS TSTR one-vs-rest ROC-AUC (left) and gap against the real-trained baseline (right).}
\label{fig:sweep_los_roc}
\end{figure}

Length-of-stay is a multi-class problem and is substantially harder than mortality. Macro-F1 (Figure~\ref{fig:sweep_los_f1}) stays in $0.16$--$0.39$ across the entire grid against a real-model baseline of $\approx 0.77$, a gap of $-0.4$ to $-0.6$ that is materially larger than the mortality gap in absolute terms. Unlike mortality ROC-AUC there is no clean monotone trend in $n_t$: the highest cells are scattered and remain within broad CI95 bands of weaker regions, suggesting seed-level fluctuations rather than genuine sharp optima. The LOS ROC-AUC heatmap (Figure~\ref{fig:sweep_los_roc}) is smoother and shows a weaker version of the mortality-AUC trend, ranging $0.11$--$0.69$ with higher values loosely clustered at moderate $n_t$, but still far from the real baseline.

Three mechanisms plausibly drive this. First, LOS categorisation depends on trajectory \emph{length} and \emph{terminal-phase dynamics}; the fixed 48-hour generation window truncates or pads every real stay to the same length and therefore destroys the primary signal the real-model classifier exploits (the stay-length-KS metric in section~\ref{sec:sweep_quality} confirms this directly). Second, LOS categories are imbalanced and macro-F1 penalises per-class collapse, so any synthetic data that under-represents short or long stays is punished more heavily than in the binary mortality task. Third, the trajectory-level classifier flattens padded sequences into a fixed-length feature vector, this is adequate when the dynamic signal is strong, but LOS depends on subtle late-sequence patterns that a linear model only partially recovers from flat features.

Practically, this means the pipeline as constituted is not a viable LOS-prediction data source at any setting on this grid. The architectural fix is not more $n_t$, it is variable-length generation or an LOS-aware post-processing step, neither of which is in scope for this thesis.

\subsection{Statistical Quality: KS, Correlation, Clinical Range Violations}
\label{sec:sweep_quality}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_quality_metrics.png}
\caption{Per-feature statistical quality metrics across the sweep. Left: average two-sample KS statistic. Middle: Frobenius norm of the difference between real and synthetic correlation matrices. Right: percentage of synthetic rows violating the clinical range table. Lower is better throughout.}
\label{fig:sweep_quality}
\end{figure}

Figure~\ref{fig:sweep_quality} is the most structurally revealing figure of the sweep and warrants close reading. The KS panel is almost flat: values sit at $0.606$--$0.643$ across the entire grid, moving by only $\approx 0.038$ between best and worst cells. A KS of $\approx 0.62$ means that for the median numeric feature, the maximum gap between real and synthetic CDFs is roughly sixty percentage points; for reference, two samples drawn from the same distribution would give $\approx 0.02$ at this sample size. The generator is therefore not close to marginal-distribution parity on most continuous features, and increasing $n_t$ or $n_{\text{noise}}$ does not help.

The correlation-preservation panel \emph{does} improve with $n_t$, dropping from a Frobenius norm of $\approx 18.4$ at $n_t{=}1$ to about $14.0$ at the best measured cell $(n_t{=}34, n_{\text{noise}}{=}2)$. This is consistent with the hypothesis that short integration paths collapse the joint onto its marginals, destroying cross-feature structure, and that additional $n_t$ steps recover pairwise relationships even when marginal fit stays weak. Again $n_{\text{noise}}$ is nearly irrelevant.

The clinical-range-violation panel is the hardest finding in this figure: violation rates sit at $52\%$--$72\%$ across every cell. More than half of synthetic rows have at least one vital sign outside the physiologic window encoded in the clinical-range table. This is not a TSTR-level problem that better classifiers can work around; it is a deployment-blocker. Two likely mechanisms: (i) the preprocessing pipeline normalises by cohort mean/std but does not clip to clinical bounds, so the flow-matching model is free to sample in the tails; (ii) the loss function has no range-awareness. The violation rate \emph{does} decrease with $n_t$ (from $\approx 72\%$ at $n_t{=}1$ to a best of $\approx 52\%$ at $(n_t{=}21, n_{\text{noise}}{=}2)$), so more integration steps soften the issue, but no populated cell reaches a deployable threshold.

\subsection{Trajectory-Level Structure and Temporal Coherence}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_trajectory_structure_metrics.png}
\caption{Trajectory-structure metrics. Left: stay-length KS statistic. Right: transition-smoothness ratio (synthetic / real mean hourly absolute change).}
\label{fig:sweep_traj_struct}
\end{figure}

The stay-length-KS panel in Figure~\ref{fig:sweep_traj_struct} is pinned at $0.98$ for every cell. This is not a bug, it is a direct consequence of the fixed 48-hour generation window documented in the Implementation chapter. Every synthetic trajectory has length~$48$; the real cohort has a continuous distribution of lengths from a few hours to several hundred. A two-sample KS on these two distributions is essentially guaranteed to be near~$1$. The figure therefore serves as a reminder that this is an architectural limitation, not a training-time one: no $(n_t, n_{\text{noise}})$ setting can ever improve this metric, and the LOS TSTR gap discussed above is partly a downstream consequence of this. The lower stay-length KS value of $0.545$ reported at the matched cell in Implementation Table~\ref{tab:backbone-trajectory} comes from a different reference cohort (the matched-cell backbone comparison uses a resampled \code{real\_test.csv} subset at the matched $(n_t{=}5, n_{\text{noise}}{=}5)$ cell rather than the full real cohort used here); both readings nonetheless agree that stay-length fidelity is a structural limitation of the fixed 48-hour window rather than a hyperparameter-tunable quantity.

The transition-smoothness ratio panel shows the opposite: strong dependence on $n_t$ and essentially none on $n_{\text{noise}}$. At $n_t{=}1$, ratios are $\approx 1.1$--$1.5$ (slightly rougher than real); by $n_t{=}13$--$21$ ratios reach $6$--$8$, meaning synthetic trajectories oscillate six-to-eight times more than real ones. This main-grid $6$--$8$ band is consistent with the matched-cell HS3F ratio of $5.44$ reported in Implementation Table~\ref{tab:backbone-trajectory}, which is measured at $n_t{=}5$, i.e.\ below the $n_t{=}13$--$21$ range and therefore on the lower, rising part of the same curve. This is counter-intuitive at first, one would expect more integration steps to produce \emph{smoother} synthetic paths. The likely explanation is that at $n_t{=}1$ the autoregressive model is essentially regressing to a conditional mean (producing artificially flat trajectories), whereas with sufficient $n_t$ the generator samples with real variance but without the temporal-smoothness prior that a real physiological trajectory has. In other words, $n_t{=}1$ is smooth because it is degenerate; higher $n_t$ is rough because the generator has no mechanism to couple successive hours. A smoothness-preserving regulariser or a larger conditioning window would address this; the current hyperparameter axes cannot.

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_temporal_correlation_metrics.png}
\caption{Temporal-correlation metrics. Left: lag-1 autocorrelation distance (feature-median gap between real and synthetic). Right: cross-feature temporal correlation drift for clinically meaningful pairs.}
\label{fig:sweep_temporal_corr}
\end{figure}

Figure~\ref{fig:sweep_temporal_corr} complements the transition panel. Autocorrelation distance is lowest ($\approx 0.10$) at moderate $n_t$ and rises toward both low and high extremes, with the clearest minimum around the $n_t{=}3$ region in marginal trends. The cross-feature correlation-drift panel is less U-shaped in the completed grid; it generally improves as $n_t$ increases and reaches its best observed value at $(n_t{=}21, n_{\text{noise}}{=}1)$. Read together, these panels indicate a trade-off within temporal coherence itself: higher $n_t$ helps cross-feature coupling but can still inflate transition roughness. The production configuration $(50, 100)$ should therefore be expected to preserve cross-feature drift better than low-$n_t$ settings while further worsening transition smoothness unless an explicit smoothness term is added.

\subsection{Privacy Metrics: DCR, Baseline Protection, Membership Inference}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_privacy_metrics.png}
\caption{Privacy diagnostics. Top row: DCR median and DCR baseline protection. Bottom row: DOMIAS-style membership inference ROC-AUC and attacker advantage.}
\label{fig:sweep_privacy}
\end{figure}

Four observations from Figure~\ref{fig:sweep_privacy}. First, the DCR median panel shows a systematic \emph{decrease} with $n_t$, from $\approx 20.4$ at $n_t{=}1$ to $\approx 19.6$ at $n_t{=}21$. The direction is expected, better-trained generators sample closer to the training manifold, but the magnitude is small: under one DCR unit over the entire grid. The exact-match-rate column in the raw CSV is $0.0$ for every cell, which confirms there is no verbatim memorisation even at the most-trained settings.

Second, the DCR baseline-protection panel is saturated at $1.0$ across the entire grid. This is the maximum score possible under the SDMetrics formulation and means synthetic rows are, on average, no closer to training rows than a uniformly sampled baseline would be. This is a strong positive privacy signal but it should be interpreted with the DCR caveat noted in the protocol section \cite{yao2025dcrdelusion}: distance-based metrics can be gamed by low-fidelity generators, and the marginal-fit results above suggest this particular signal may partly reflect the generator's overall distributional drift rather than a principled privacy guarantee.

Third, the DOMIAS-style membership-inference ROC-AUC sits at $0.594$--$0.599$ across the grid, with a slow monotone creep upward as $n_t$ grows. A value in this band corresponds to an attacker with a small but non-trivial advantage over random guessing; more precisely, if a row is presented to the attacker, they are correct about training membership $\approx 60\%$ of the time. The attacker-advantage panel is $0.0$ everywhere because the attacker advantage is reported as $\max(0, 2\cdot\text{AUC}-1)$ thresholded against the discrete seed-level sampling variance, and at this scale the thresholded estimate is zero even when the raw AUC is $0.6$. This discrepancy is worth flagging in future reporting: advantage and AUC are not reading the same story here, and the AUC number is the more honest single summary.

Fourth, privacy metrics are the \emph{only} family in the sweep where the trend with $n_t$ is monotonically \emph{unfavourable}: utility improves (or saturates) with $n_t$, quality improves with $n_t$, but DCR shrinks and membership-inference AUC grows. This is the classical utility--privacy trade-off materialised in one figure. At production scale $(n_t, n_{\text{noise}}){=}(50, 100)$, both effects should extend in the same direction.

\subsection{Training Cost and Scaling}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_training_time.png}
\caption{Total training time (hour-0 module + autoregressive module, in minutes) as a function of $n_t$ and $n_{\text{noise}}$.}
\label{fig:sweep_time}
\end{figure}

Figure~\ref{fig:sweep_time} shows cost scaling. The right panel is nearly a pencil of straight lines through the origin: for each fixed $n_t$, training time is approximately linear in $n_{\text{noise}}$, as expected because the noise-anchor loop is the inner-most loop in the training step. The left panel is super-linear in $n_t$ for large $n_{\text{noise}}$ but approximately linear for small $n_{\text{noise}}$. A bilinear least-squares fit on the completed grid gives $T \approx 50.87\,n_t n_{\text{noise}} - 102.06\,n_t - 101.65\,n_{\text{noise}} + 642.54$ (seconds), with $R^2{=}0.901$. The dominant term is still $n_t n_{\text{noise}}$, but the lower $R^2$ on the full grid highlights stronger non-linear overheads in the high-cost corner. The split view \code{sweep\_training\_time\_split.png} confirms that this departure comes from the autoregressive module, not from hour-0.

Two consequences. First, the most expensive measured cell is now $(n_t, n_{\text{noise}})=(34,34)$ at $\approx 18.9$ hours ($68{,}037$\,s). Second, the \emph{marginal} cost of increasing $n_t$ at high $n_{\text{noise}}$ is not rewarded by proportional utility gains: going from $n_t{=}21$ to $n_t{=}34$ in the high-noise corner adds tens of thousands of seconds but only moves marginal-mean ROC-AUC from $\approx 0.629$ to $\approx 0.638$, a change inside broad seed-level CI95 overlap.

\subsection{Uncertainty Structure}

\begin{figure}[h]\centering
\includegraphics[width=\textwidth]{sweep_uncertainty_heatmaps.png}
\caption{CI95 half-widths across the sweep for each reported metric. Darker cells indicate wider confidence intervals.}
\label{fig:sweep_uncert}
\end{figure}

Figure~\ref{fig:sweep_uncert} reports the CI95 half-widths across the grid. Three patterns are worth noting. The mortality ROC-AUC CI95 is large ($\pm 0.06$ to $\pm 0.22$) and is widest at small $n_t$, narrowing as the classifier moves away from chance; this means the left column of the utility heatmaps should not be read point-by-point. LOS macro-F1 CI95 is similarly wide ($\pm 0.14$ to $\pm 0.47$ in some cells), reinforcing the earlier point that LOS peaks are seed noise rather than genuine optima. In contrast, the quality-metric CI95s (KS, correlation, range violations) are extremely tight ($\pm 0.0003$ to $\pm 0.06$), because these metrics aggregate over many feature-pairs and many rows and so average out the trajectory-sampling variance. This validates treating the quality panel trends as real rather than sampling artefacts.

The matched-cell three-seed protocol is useful for uncertainty awareness, but it is still too few-shot to tightly pin TSTR utility differences between close backbones. A higher seed count at production-scale cells would materially improve inferential strength.

Because the prediction depends sensitively on the assumed functional form, three plausible saturating models are fit to the $n_t$-marginalised ROC-AUC series from the fully populated grid and report all three:

\begin{itemize}
    \item Exponential saturation $y = a - b\,e^{-c\,n_t}$: asymptote $a \approx 0.633$, so $\hat{y}(n_t{=}50) \approx \hat{y}(n_t{\to}\infty)$.
    \item Power decay $y = a - b\,n_t^{-c}$: $\hat{y}(n_t{=}50) \approx 0.654$, $\hat{y}(n_t{=}100) \approx 0.668$.
    \item Logarithmic $y = a + b\log n_t$: $\hat{y}(n_t{=}50) \approx 0.667$, $\hat{y}(n_t{=}100) \approx 0.695$.
\end{itemize}

The three forms agree closely on the measured grid and diverge only in the extrapolation region. The exponential fit best matches the observation that marginal-mean ROC-AUC moved by only $\approx 0.009$ between $n_t{=}21$ and $n_t{=}34$; the logarithmic fit is optimistic because it assumes unbounded (if slow) growth. The honest middle estimate for production mortality ROC-AUC is therefore still around $\hat{\text{ROC-AUC}} \approx 0.65 \pm 0.02$. Under no form does the extrapolation close the gap to the real-trained baseline ($0.82$).

$n_{\text{noise}}$ extrapolation is simpler: the $n_{\text{noise}}$-marginalised curve is flat within seed noise across the populated grid, so $n_{\text{noise}}{=}100$ is not expected to change utility by more than a few thousandths. The same logic applies to quality and privacy metrics.

\paragraph{What the matched cell says about the CTGAN baseline.}
CTGAN's mortality ROC-AUC of $0.497$ is indistinguishable from a constant-score classifier, the standard interpretation of AUROC $\approx 0.5$, and its balanced accuracy ($0.507 \pm 0.020$) is similarly near chance. In the raw-accuracy column of the underlying CSV, CTGAN scores $0.622$, essentially tied with ForestFlow's $0.637$; however, under MIMIC's mortality class imbalance this raw figure mostly reflects majority-class (survived) prevalence rather than discrimination, which is exactly why Table~\ref{tab:backbone-matched} reports the balanced form. Once the majority-class bias is removed, CTGAN's classifier is no better than chance at ranking patients by mortality risk, despite its average-KS being the best of the three ($0.686$). The baseline is therefore failing for the right reason: it reproduces approximately-correct feature marginals (consistent with its best-in-class KS).

It does not, however, preserve the lagged-feature-to-outcome dependency structure a downstream classifier needs. This is consistent with the adapter being trained on the transformed joint table and sampled without per-row autoregressive conditioning, as flagged in the section \textbf{Baseline Scope and Interpretation} disclaimer. The trajectory-level metrics reinforce this reading: CTGAN's transition smoothness ratio ($9.86$) is roughly double that of the tree-based backbones, so hour-to-hour transitions are correspondingly jumpier, a direct consequence of the missing per-row conditioning. The baseline is therefore informative in exactly the way a baseline should be: it demonstrates that the utility signal captured by patient-level TSTR ROC-AUC (and, equivalently, by balanced accuracy once calibration noise is absorbed into score ranking) is not trivially attainable by a generator that merely produces near-correct marginals.

\paragraph{CTGAN hyperparameters and fairness of the comparison.}
The CTGAN baseline is configured with the hyperparameters reported in the original paper \cite{xu2019ctgan} and exposed as library defaults in the \code{ctgan} package: a two-layer $256\text{-}256$ generator MLP, a two-layer $256\text{-}256$ discriminator MLP, embedding dimension $128$, $300$ training epochs, batch size $500$, Adam learning rate $2 \times 10^{-4}$ with weight decay $10^{-6}$, and PAC size $10$ (see \code{scripts/run\_sweep.py}, \code{CTGAN\_DEFAULT\_*}, exposed to \code{run\_backbone\_comparison.py} through the \code{--ctgan-*} CLI flags). Two small reproducibility points are worth noting: (i) when the requested PAC does not divide the batch size the adapter silently falls back to the largest valid divisor (no effect at $(500, 10)$), and (ii) PyTorch and OMP thread pools are pinned to a single thread via \code{torch.set\_num\_threads(1)} and \code{OMP\_NUM\_THREADS=1} so the training-time comparison is not biased by CTGAN exploiting more CPU threads than the XGBoost-based backbones. No CTGAN-specific tuning was attempted beyond the published defaults, so the comparison is against a standard out-of-the-box baseline rather than a hand-tuned one; if anything, this biases the comparison in CTGAN's favour by removing the possibility that the observed collapse is caused by hyperparameter misconfiguration.

\paragraph{Compute cost versus utility.}
The most striking feature of the matched-cell comparison is the inversion between computational cost and downstream utility. CTGAN is the most expensive model to train by a large margin: $8017$~s total wall-clock at $(n_t{=}5, n_{\text{noise}}{=}5)$, compared with $3224$~s for ForestFlow ($\approx 2.5\times$ slower than ForestFlow) and $2194$~s for HS3F ($\approx 3.7\times$ slower than HS3F). Despite consuming more compute than the two tree-based backbones combined, CTGAN produces the worst downstream utility on every TSTR axis examined at this cell: mortality ROC-AUC ($0.497$, significantly below HS3F's $0.672$ at three seeds), LOS macro-F1 ($0.173$, roughly $40\%$ lower than the $\approx 0.28$ achieved by both tree-based backbones), and transition smoothness ratio ($9.86$, roughly $2\times$ the HS3F/ForestFlow value). Balanced mortality accuracy is numerically close across all three backbones ($0.501$--$0.535$), but this near-chance clustering is itself the critique: the extra CTGAN wall-clock buys no measurable discrimination improvement over the cheaper tree-based alternatives. The longer training time is therefore not purchasing any measurable utility advantage. This reinforces rather than qualifies the ``Why HS3F'' decision developed below: HS3F wins the matched cell on ranking utility, trajectory coherence, training cost, and privacy parity, and the closest neural baseline at standard hyperparameters loses decisively on every utility axis despite consuming nearly four times the wall-clock of HS3F.

\paragraph{Interpreting the HS3F--ForestFlow discrimination profile.}
On the balanced-accuracy row of Table~\ref{tab:backbone-matched}, all three backbones cluster near chance: HS3F $0.501 \pm 0.012$, CTGAN $0.507 \pm 0.020$, ForestFlow $0.535 \pm 0.048$. The nominal ForestFlow advantage overlaps HS3F's confidence interval and is not statistically significant at three seeds. On ROC-AUC the ordering reverses, HS3F $0.672$ vs.\ ForestFlow $0.619$, and with CI95 half-widths of $\approx 0.021$ (HS3F) and $\approx 0.077$ (ForestFlow) from \code{backbone\_matched\_comparison.csv} the tree-based gap is again not statistically significant, while the HS3F--CTGAN gap on ROC-AUC is. The honest reading of the mortality row is therefore: (1) no backbone produces a downstream classifier that reliably discriminates mortality under the balanced-accuracy lens; (2) HS3F's ROC-AUC lead over CTGAN is real, its ROC-AUC lead over ForestFlow is suggestive but not significant at three seeds; (3) LOS macro-F1 is statistically tied between HS3F and ForestFlow ($0.284$ vs.\ $0.283$) and significantly above CTGAN ($0.173$). Patient-level utility at this cell is therefore an \emph{incremental} win for HS3F on ranking metrics rather than a categorical one; the decisive separation between HS3F and ForestFlow appears in the trajectory-level metrics discussed next.

\paragraph{Where HS3F does separate from ForestFlow: temporal coherence.}
The matched-cell trajectory-level metrics in Table~\ref{tab:backbone-trajectory} separate the two tree-based backbones more decisively than the patient-level TSTR row does. HS3F's autocorrelation distance ($0.22$) is roughly half of ForestFlow's ($0.44$), and its temporal cross-feature correlation drift ($0.26$) is likewise the lowest of the three. Both metrics are direct measurements of whether per-patient lag-1 temporal structure is preserved, both are computed on the same matched-cell synthetic cohort, and both use the same three-seed averaging, so the comparison is not confounded by sampling differences. The plausible mechanism follows from the backbone design. HS3F generates intervention flags (e.g.\ ventilation, vasopressors) and categorical admission variables through explicit classifier heads, so at step $j+1$ of the feature-wise autoregressive chain the conditioning vector contains cleanly-typed previously-generated features; ForestFlow transports the same discrete features as continuous flows and rounds post-hoc, so the conditioning vector that drives transition generation contains continuous intermediate values that round-trip through the discretization boundary. A $2\times$ increase in autocorrelation distance for ForestFlow, measured on identical patients and seeds, is exactly the signal the design hypothesis predicts this round-trip would produce. This is the strongest positive evidence for HS3F's architectural choice on our data.

\paragraph{Privacy verdict.}
All three generators achieve DCR baseline-protection $= 1.0$ under the SDMetrics ceiling, zero exact-match synthetic records (\code{dcr\_exact\_match\_rate} $= 0$), and MIA ROC-AUC $\approx 0.497$ (indistinguishable from chance at three seeds). No configuration shows gross memorisation at this cell, and the privacy picture is effectively a tie across HS3F, ForestFlow, and CTGAN; the compute-cost picture is separately covered in the paragraph above.

\paragraph{Why HS3F is the primary backbone for the main grid.}
Taken together, the matched-cell evidence at $(n_t{=}5, n_{\text{noise}}{=}5)$ is what motivates using HS3F as the primary backbone for the main hyperparameter sweep:
\begin{itemize}\tightlist
    \item \textbf{Temporal coherence.} Two of the four trajectory-level metrics (autocorrelation distance, cross-feature drift) favour HS3F by roughly $2\times$ over ForestFlow; the other two are tied or within seed noise.
    \item \textbf{Ranking utility.} HS3F's mortality ROC-AUC ($0.672$) is the highest of the three; under three seeds the HS3F--ForestFlow gap is not statistically significant, but the HS3F--CTGAN gap is. LOS macro-F1 is tied with ForestFlow.
    \item \textbf{Architectural alignment to the feature schema.} HS3F's explicit categorical heads are designed for a mixed continuous/binary/categorical feature set (binary intervention flags, categorical admission codes). ForestFlow's all-continuous flow requires post-hoc clipping of discrete features, and the measured $2\times$ increase in autocorrelation distance is consistent with this round-trip introducing hour-to-hour transition noise.
    \item \textbf{Computational cost.} HS3F trains $\approx 32\%$ faster than ForestFlow and $\approx 3.7\times$ faster than CTGAN at this cell. Across a $64$-cell grid this compounds into a decisive sweep-feasibility advantage.
    \item \textbf{Privacy parity.} Proximity (DCR) and membership-inference (DOMIAS-style) diagnostics are indistinguishable across backbones, so HS3F's utility and cost advantages are not bought at a measured privacy cost.
\end{itemize}

\paragraph{Caveats on the backbone decision.}
The backbone-axis evidence rests on a single matched $(n_t, n_{\text{noise}})$ cell with three trajectory seeds. The HS3F main grid is now complete; the ForestFlow and CTGAN full grids are not run. The HS3F--ForestFlow ROC-AUC gap at this cell is not statistically significant, and the backbone ordering on other cells may differ. Main-grid conclusions in this chapter are therefore HS3F-specific by construction; claims transfer to ForestFlow or CTGAN only under the matched-cell comparison. Extending the three-seed matched comparison to the full grid is flagged in \code{docs/project/status.md}; the CSV-level artefacts (\code{results/backbone\_comparison/sweep\_results\_\{hs3f,forest\_flow\_copy,ctgan\}.csv}) already carry the full schema and are ready for the next regeneration pass.

\subsection{Synthesis and Implications}

The main-grid HS3F sweep yields five defensible findings.

First, \textbf{$n_t$ dominates utility but exhibits diminishing returns}. Marginal-mean ROC-AUC gains $\approx 0.10$ between $n_t{=}1$ and $n_t{=}8$, only $\approx 0.012$ more between $n_t{=}8$ and $n_t{=}21$, and a further $\approx 0.009$ at $n_t{=}34$. The curve is still inching upward but the per-unit-compute yield is collapsing; doubling or quadrupling $n_t$ beyond $n_t{\approx}8$ is a small-gain/large-cost bet, and the three saturating fits agree the residual gap to the real baseline will not close via $n_t$ alone.

Second, \textbf{$n_{\text{noise}}$ does not materially improve any utility, quality, or privacy metric on this grid}. Its only monotone effect is on training time. Setting $n_{\text{noise}}$ to $100$ rather than a small value is therefore hard to justify on evidence; in fact, reducing $n_{\text{noise}}$ to the single-digit range would save roughly an order of magnitude of compute without measurable TSTR loss. The production choice of $n_{\text{noise}}{=}100$ should be revisited with this evidence.

Third, \textbf{two structural limitations of the architecture bound what the hyperparameters can achieve}: the fixed 48-hour generation window pins stay-length KS at $\approx 0.98$ and damages LOS TSTR at every setting, and the absence of a trajectory-smoothness prior causes the transition-smoothness ratio to \emph{degrade} as $n_t$ grows. Neither can be fixed by tuning $n_t$ or $n_{\text{noise}}$.

Fourth, \textbf{clinical-range violation is the most serious single finding in the sweep}. Roughly $52\%$--$72\%$ of synthetic rows contain at least one out-of-range vital sign, and the rate only partly improves with $n_t$. For any downstream application that requires plausible individual records (not merely plausible aggregates), the pipeline as it stands would need a post-generation clipping or rejection stage, or a loss-level range penalty.

Fifth, \textbf{the privacy picture is cautiously positive but deserves care}. Exact-match rate is zero, DCR baseline protection is saturated at $1.0$, and membership-inference AUC is $\approx 0.60$ (above random but well below the $0.7$--$0.8$ bands typically cited for clear leakage). However, DCR can be flattered by overall distributional drift, the same drift the quality metrics make visible, so the privacy advantage is partly a side-effect of the utility shortfall. A meaningfully higher-utility generator would likely show a meaningfully worse privacy profile; the utility--privacy trade-off should be re-measured at that operating point.

Taken together, the evidence supports running the production configuration as a single validation run but does not support it as an efficient operating point. A better-justified production cell, under the assumptions of the current architecture, would sit near $(n_t, n_{\text{noise}}){\approx}(8, 2)$, well inside the diminishing-returns region on utility, near the temporal-coherence sweet spot, and roughly two orders of magnitude cheaper to train than $(50, 100)$. Further gains beyond that would most likely come from architectural changes (variable-length generation, range-aware loss, smoothness regulariser) rather than additional grid search. The backbone comparison above supports HS3F as the fixed backbone for any such operating point.

\subsection{Goal Status: Met, Partial, Not Met}

Against the dissertation objectives and acceptance targets, the evidence in this chapter supports the following judgement:
\begin{itemize}\tightlist
    \item \textbf{Met (research feasibility):} Tree-based flow matching can be extended to sequential ICU synthesis with a working two-stage system and measurable utility transfer above chance.
    \item \textbf{Partially met (utility):} Ranking utility improves with $n_t$ and reaches ROC-AUC $0.659$, but the predefined ``within 0.15 ROC-AUC'' target is still missed against the real baseline ($\approx 0.82$).
    \item \textbf{Not met (minority retention):} Mortality and LOS F1 remain low across the grid, indicating persistent minority-class collapse.
    \item \textbf{Not met (practical adequacy):} Range violations remain high (best $\approx 52\%$) and stay-length fidelity is structurally poor under fixed-length rollout, so practical release readiness is not demonstrated.
    \item \textbf{Partial and diagnostic only (privacy):} No exact matches and bounded DOMIAS-style AUC provide useful diagnostics, but they do not establish formal privacy release assurance.
\end{itemize}

%==============================================================================
\section{Threats to Validity}
\label{sec:threats-to-validity}
%==============================================================================

To contextualise the sweep findings above, this section states the main validity limits that bound interpretation and transferability.
The preceding empirical claims are bounded by the following validity threats, grouped per construct/internal/external/conclusion validity \cite{runesonHost2009CaseStudyGuidelines}.
The items below are stated directly: they bound what the current implementation and sweep can claim.

\subsection{Construct validity}

\paragraph{Operationalization of ``utility'' and ``fidelity''.}
Multi-task TSTR (logistic baselines in \code{tstr\_framework}) and the sweep's marginal/correlation/range metrics are \emph{proxies} for clinical usefulness and distributional faithfulness.
They still omit temporal consistency across full stays, calibration of predicted risks, and expert review of clinically implausible joint patterns.
\textbf{Mitigations in code:} two complementary patient-level tasks (mortality, LOS) reduce single-score overfitting; KS, $\|\Delta C\|_F$, and range violations deliberately stress different failure modes; post-decode clipping is intended to align known flags with binary semantics (Implementation chapter, section \textbf{Post-Decode Clinical Normalisation}); and privacy is now monitored with DCR-based diagnostics plus DOMIAS-style MIA scoring (Implementation chapter, section \textbf{Privacy Risk Metrics}).
\textbf{Gaps:} no multivariate two-sample tests, no clinician validation loop, and no full adaptive/shadow-model MIA battery beyond the nearest-neighbour DOMIAS-style attack proxy. Additionally, binary intervention flags (\code{vent}, \code{vaso}) are transported as continuous flows during HS3F generation and rounded post-hoc; the fraction of generated values that fell outside $\{0,1\}$ before clipping is not currently tracked, leaving one source of KS inflation and plausibility degradation unmeasured.

\paragraph{Label and prevalence semantics.}
Encoded categories and rare events may dominate apparent TSTR scores (e.g., accuracy under imbalance).
\textbf{Mitigations:} macro-F1 and ROC-AUC on LOS; explicit discussion of F1 collapse in the sweep results (Figure~\ref{fig:sweep_mort_f1}).
\textbf{Gaps:} prevalence-aware baselines and stratified reporting are not yet first-class in the sweep output.

\subsection{Internal validity}

\paragraph{Confounding by configuration and compute.}
The sweep varies both $n_t$ and $n_{\text{noise}}$, but wall-clock differences (e.g., training time varies from $\approx 3$ minutes to $\approx 18.9$ hours in the most expensive cell) can couple ``better'' metrics with unequal optimisation budgets unless training is stopped by a common criterion.
\textbf{Mitigations:} per-stage timings and degeneracy annotation are logged; interpretation already warns against ranking on this CSV alone.
\textbf{Gaps:} no matched early-stopping or compute-matched comparison protocol in the sweep.

\paragraph{Determinism versus accidental reuse.}
Seeds and artefact paths support repeat runs, but the transformed-data cache (\code{\_build\_cache\_signature}, \code{\_load\_transformed\_cache}) can silently reuse large arrays when signatures match, correct when inputs are unchanged, hazardous if a user assumes a fresh transform after a partial edit that does not bump mtime or path.
\textbf{Mitigations:} signature includes path, size, mtime, column names, and index vectors; cache miss overwrites.
\textbf{Gaps:} \code{run\_sweep.py} exposes \code{--refresh-transformed-cache}, but cache-refresh decisions are still manual; semantic changes that preserve file metadata can be missed without explicit operator checks.

\paragraph{Evaluation leakage and split stability.}
TSTR and quality metrics depend on how real/synthetic splits are formed; any overlap between training generator inputs and evaluation cohorts would inflate utility.
\textbf{Mitigations:} the chapter ties evaluation to the implemented split conventions in \code{tstr\_framework} and preprocessor indices.
\textbf{Gaps:} formal independence guarantees across hour-0, autoregressive training, and TSTR splits should be verified explicitly for each experiment script version.

\subsection{External validity}

\paragraph{Single database and pipeline.}
Results apply to MIMIC-derived hour-0 plus autoregressive ICU rows, the chosen feature set, and the Forest-Flow/HS3F backbones implemented in this work, not to ward workflows, other charting systems, or primary-care EHRs.
\textbf{Mitigations:} two-stage design and backbone selection document the scope of the generative family.
\textbf{Gaps:} cross-hospital replication on the full MIMIC cohort is not attempted; preprocessor and item definitions remain cohort-specific. A cross-dataset replication on the public PhysioNet Challenge 2012 benchmark (section~\ref{sec:challenge2012-eval}) is implemented and currently executing; its numerical results are out of scope for this submission.

\paragraph{Synthetic sample size and conditioning.}
Conditional generation reuses real conditioning rows in sweep mode; this anchors context but limits exploration of counterfactual trajectories and may hide defects in the conditional model when the condition distribution is narrow.
\textbf{Gaps:} ablations over synthetic cohort size and purely synthetic conditioning chains are future work.

\subsection{Conclusion validity}

\paragraph{Statistical power and multiplicity.}
The expanded sweep (currently $64$ completed configurations) repeats each cell across three trajectory seeds and records mean, sample standard deviation, and $95\%$ confidence-interval half-widths for key utility, quality, and privacy metrics.
Scanning several metrics and tasks still raises multiplicity concerns, and with only three seeds per cell the intervals should be treated as uncertainty indicators rather than definitive significance tests.
\textbf{Mitigations:} resumable \code{sweep\_results.csv}, failure rows preserved, uncertainty columns exported, and overlap-aware interpretation rather than deterministic point-estimate ranking.
\textbf{Gaps:} larger seed counts, bootstrap CIs, and multiple-comparison discipline are still future work in \code{run\_sweep.py}.

\paragraph{Threshold-based degeneracy.}
\code{degenerate\_flag} encodes a fixed heuristic on near-random utility; it reduces false confidence but can miss structured failure (good average AUROC with pathological calibration) or flag borderline runs arbitrarily.
\textbf{Mitigations:} flagged rows are retained, not dropped.
\textbf{Gaps:} sensitivity analysis for thresholds and correlation with distributional metrics is outstanding.

\section{Cross-Dataset Replication on Challenge 2012}
\label{sec:challenge2012-eval}
%==============================================================================

To probe pipeline portability beyond MIMIC-III, the implementation additionally targets the open-access PhysioNet Challenge 2012 set-A cohort (4{,}000 ICU patients, 48 hourly bins); the pipeline wiring is documented in section~\ref{sec:challenge2012-impl}. The same sweep grid, TSTR tasks, fidelity metrics, trajectory coherence metrics, and privacy metrics are applied, without any re-implementation, via \code{scripts/run\_sweep.py --dataset challenge2012}.

\paragraph{Status at time of submission.}
The Challenge 2012 sweep is in progress and intentionally not included in the numerical tables and figures of this chapter. Reporting its outcomes before sweep completion would risk partial-cell results distorting per-metric comparisons with the MIMIC sweep; the unified schema (\code{results/sweep\_results.csv}) is designed to accommodate the new runs without retrofitting the analysis pipeline.

\paragraph{What this benchmark does and does not establish.}
The Challenge 2012 run is a reproducibility supplement, not part of the dissertation's evidential core. It shows that the same generator, preprocessor, and evaluation stack can be executed end-to-end on a semantically adjacent open cohort (static demographics + hourly vital medians + mortality/LOS labels). The core empirical claims in this chapter are based on MIMIC-III. Challenge 2012 is therefore reported as methodological portability support rather than as evidence for clinical generalisation; cross-hospital or cross-speciality validity would require additional cohorts beyond scope.

\paragraph{Metric-reading caveats for Challenge 2012.}
Two implementation-dependent caveats should be stated explicitly for interpretation. (i) In the current Challenge pipeline, both real and synthetic tables are represented as fixed 48-hour windows; therefore \code{stay\_length\_ks} is expected to be near $0$ and is not a discriminative indicator of variable-length trajectory fidelity in this setting. (ii) Elevated \code{range\_violation\_pct} values on Challenge can partly reflect implausible measurements already present in the real cohort (notably blood-pressure zeros/outliers), so this metric should be read as a combined signal of source-data quality and generator behaviour rather than as a pure model-failure score.
